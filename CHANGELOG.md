# Changelog

## v0.2.4

- **Breaking / Bugfix:** a `UNIQUE` index that `AutoMigrate` cannot create now fails the migration instead of being logged and ignored (roadmap 10.3). **This can block a boot that previously succeeded — see the upgrade note below.** Previously a failed unique index left the model declaring a constraint the database was not enforcing: `mfx:"unique"` promised a guarantee that did not exist, and every duplicate that should have been refused was accepted silently, with nothing to surface it until something downstream tripped over the data. This affected the HMAC index behind `mfx:"encrypted,unique"` fields and any `ModelConfig.Indices` entry with `Unique: true`. Note the inconsistency it resolves: the `ALTER TABLE ADD COLUMN` immediately beside the HMAC index already returned an error — only the index had been downgraded to a warning. A failed **plain** index is unchanged and still only warns: a missing one costs a table scan, not a constraint. **Also fixed:** the HMAC index creation sat inside the "hmac column is missing" branch, so a boot that added the column but failed to build the index never retried it — every later boot saw the column, skipped the block, and left the constraint permanently absent. It is now ensured on every migration (`CREATE ... IF NOT EXISTS`, so the retry is free). **Upgrade note:** because of that retry, a deployment that has silently accumulated duplicate values in an `encrypted,unique` field will now fail to start, with the table, column and remedy named in the error. That data violates a constraint the model already declared; de-duplicate it and start again.

- **Feature:** `maniflex.RequiresField` declares the model fields a middleware gates, so a misspelt field name fails at startup instead of failing open (roadmap 10.2). A `validate.RestrictField("subscription_expire_at", ...)` with a typo is inert — it watches for a body key nothing sends, while the real field keeps its name and nothing gates it. Nothing at runtime can catch this: from inside a request, "this model has no such field" is indistinguishable from a gate deliberately registered across models where only some carry it. Only the registration knows which it is, so that is where the declaration goes: `maniflex.RequiresField("subscription_expires_at")` alongside `ForModel`. The `ForModel` scope makes the check exact rather than a guess — with it, every named model must have every declared field; without it, at least one registered model must, since a gate no model can trigger cannot be doing anything. The declared name is checked against the **model**, not against the middleware's own argument, so writing it twice catches a misspelling rather than duplicating one. It is a general `MiddlewareOption`, usable by any middleware that gates a field by name (`response.RedactField`, hand-written gates), and it is opt-in — a registration that declares nothing is not checked, so existing code is unaffected. `validate.RestrictField` keeps its first-request warning as a fallback for undeclared registrations. Violations join the aggregated startup report from 10.1.

- **Breaking / Feature:** startup validation and `Config.Strict` (roadmap 10.1). Five configuration mistakes that used to warn and then silently discard what you wrote are now hard failures. **This will break misconfigured applications on upgrade — deliberately, since those applications are already not doing what their author wrote.** They are: a `ModelConfig` at argument position 0 or two in a row (the config was dropped entirely, so a `ModelConfig{Headless: true}` silently did not apply); an invalid `mfx:"scheduled"` tag (the field was dropped, so the sweep never ran); `mfx:"relation"` on a field not ending in `ID` (the target was inferred from the whole field name, almost never a real model — write `mfx:"relation:Target"`); and a middleware registered on a pipeline step its operations never reach (frozen into the chain and never run, which for an authorisation check is a silent hole). The first three are registration errors from `Register`; the last two need the whole registry and surface when the router is built. Three of these had promised a fix in their own warning text for several releases. The new `Config.Strict` gates only the warnings that have a legitimate reading: a relation whose target is unregistered (it may be a plain foreign id), the standalone `/files` endpoints mounted without auth middleware, and a `Config.StaticDir` that does not exist. Turn it on in CI and staging. Every problem found is reported **together, in one message**, with strict-only entries marked `(Config.Strict)` — a fix-one-restart-repeat loop becomes one pass. Validation is also **hoisted ahead of migrations and service startup**, so a configuration error can no longer be discovered after the schema has already been altered; `Start`/`StartWithContext` return it, while `Handler()` still panics as it has no error return. Warnings that describe legitimate operational states — an orphan DB column, column type drift, chained scheduled transitions — are unchanged. See `docs/src/reference/strict-mode.md`.

- **Feature:** soft-delete restore endpoint (roadmap 5.19). `ModelConfig.RestoreEnabled` mounts `POST /:model/{id}/restore`, clearing the delete marker so the row is live again — `200` with the restored record, `404` if there is no such row or it is not deleted (mirroring the re-delete guard). Previously the docs sent you to a custom action with raw SQL, because a soft-deleted row is invisible to every read path and unreachable by `Update`: the adapter's soft-delete condition is applied unconditionally at all eleven read/update sites, and the marker columns are `readonly`. Restore is therefore its own statement, the mirror image of the delete. It is **off by default** — un-deleting is privileged, and an endpoint appearing merely because a version was upgraded would not be covered by the authorisation an app already wrote — and is only mounted for models that actually soft-delete. It **dispatches as `OpUpdate`**, so every middleware already registered for "who may modify this row" (auth, tenancy, force filters, audit) governs it unchanged, with no new operation constant to discover; `ctx.IsRestore()` tells the two apart where it matters, and the request carries no body. Scoping is enforced by pushing the request's forced filters into the restore statement, since the target cannot be read back to check it — a caller cannot un-delete another tenant's row by knowing its id. Only the marker is written: `updated_at` is left alone so a restore does not read as an edit. Adapters opt in through a new optional `maniflex.Restorer` interface (bundled SQLite and Postgres implement it; a third-party adapter that does not answers `501 RESTORE_UNSUPPORTED`), so nothing breaks. Cascade is not undone — nothing records which children an `onDelete:cascade` removed.

- **Feature:** token revocation and logout (roadmap 12.1). A JWT was valid until its `exp` and nothing could take it back, so a logout, a password change, or a compromised account left every outstanding token working until it aged out. Setting the new `JWTOptions.Revoker` makes the JWT middleware (both `JWTAuth` and `JWKSAuth`) consult a blocklist after the signature and registered claims check out. It revokes at two granularities, and both are needed: one token by its `jti` (`401 TOKEN_REVOKED`), and every token of a user issued before a cutoff — "log out everywhere", where the outstanding `jti` values are unknown and unknowable. Two ready-to-mount actions come with it: `auth.Logout(rev, path)` revokes the calling request's own token and `auth.LogoutAll(rev, path, retain)` revokes all of the caller's, both answering `204`. Backends: `auth.NewMemoryRevoker()` in-process (single replica; a restart un-revokes everything), and a new `middleware/auth/redis` submodule for shared, restart-surviving state. Switching revocation on makes a `jti` mandatory — a token without one cannot be revoked, so it is refused with `401 TOKEN_NOT_REVOCABLE` rather than granted a permanent exemption from the blocklist. Lookups **fail closed**: `Revoker` is its own error-returning interface rather than a reuse of `maniflex.CacheStore` (whose `Get` reports a miss and an outage identically), and an unreachable store answers `503 REVOCATION_UNAVAILABLE` instead of quietly treating every revoked token as valid again. Also new: `JWTOptions.VerifyToken`, a hook at the same point for issuer-specific rules, which may refuse the token or enrich the principal. Both options default to nil, so an existing `JWTAuth` is unchanged and costs no per-request lookup.

- **Feature:** both download routes — `GET /files/{key...}` and the per-model attachment route — now honour the HTTP `Range` request header, so an interrupted download can resume and a client can seek into a large media file instead of refetching it (roadmap 3B.3b). A satisfiable range answers `206 Partial Content` with `Content-Range` and a window-sized `Content-Length`; `Accept-Ranges: bytes` is advertised on full responses too, but only when the backend can really serve a window. A well-formed range starting past the end of the object is `416 RANGE_NOT_SATISFIABLE` with `Content-Range: bytes */<size>`; a malformed header, an unknown unit, or a multi-range request (which would need a `multipart/byteranges` body) is declined and served whole with `200`, as the spec permits. The security headers of a full download — `nosniff` and the inline/attachment decision — are identical on a `206`. Backends opt in through a new optional `maniflex.RangeRetriever` interface (`RetrieveRange(ctx, key, offset, length)`), implemented by `LocalStorage` (seeks the file) and `storage/s3` (pushes the window into `GetObject`, so bytes the client did not ask for never leave the bucket or the egress bill). It is additive, not a `FileStorage` change: a third-party backend that does not implement it keeps working unchanged, falling back to `net/http`'s `ServeContent` when its reader is seekable and otherwise ignoring the header. Implementations never parse a header or clamp a window — the framework resolves the range against the size `Stat` reports first, so `offset` and `length` arrive absolute and in bounds.

- **Feature:** `mfx:"relation:Parent;onDelete:cascade|setNull|restrict"` is now enforced — the roadmap's cascading-deletes item (5.16). The tag parsed and was stored on the relation, but nothing ever read it: a `DELETE /authors/{id}` left every `Post.author_id` dangling, a protective-looking directive that did nothing. It is enforced two ways, chosen per relation. **When neither side soft-deletes**, a real database `FOREIGN KEY … ON DELETE …` constraint carries it (declared inline at `CREATE TABLE` on SQLite, which permits forward references and has `foreign_keys` on; added by an idempotent `ALTER TABLE` pass on Postgres) — the database cascades, nulls, or restricts natively, and also enforces referential integrity on insert. **When either side soft-deletes**, the database cannot help — a soft delete is an `UPDATE`, so `ON DELETE` never fires, and a DB cascade can only hard-delete — so maniflex enforces it in the delete's own transaction: a soft-delete child of a cascaded parent is soft-deleted _identically_ (its `deleted_at` is set, the row kept), `cascade` recurses (cycle-guarded) through maniflex's own `Delete`, `setNull` nulls the FK, and `restrict` refuses with `409 DELETE_RESTRICTED`. Either path is atomic — a `restrict` that fires rolls back a sibling `cascade`. Validated at startup: an `onDelete` must target a registered model, and `setNull` needs a nullable (pointer) FK, else a registration error. A DB-enforced `restrict` on delete and a bad-parent insert both normalise to `409` (the tx-delete path now normalises constraint errors like create/update already did). Note SQLite cannot add a FK to an existing table, so new `onDelete` constraints land only on freshly-created SQLite tables; the soft-delete path is app-enforced regardless.
- **Feature:** new `mfx:"file,upload:stream"` pipes a multipart upload straight to storage as it arrives off the socket, instead of buffering the whole request to the app server's disk first — the completion of roadmap 3B.2a. The default multipart path drains the body before the handler runs (in memory up to `MaxUploadMemory`, the rest spooled to temp files), so a 5 GB upload lands on the app server in full and is then copied to the backend; streaming removes that landing. It is opt-in per field and the request shape is unchanged — the common case (no `upload:stream` field) keeps the untouched `ParseMultipartForm` path, so existing uploads are unaffected. `POST /files` now always streams. Because a stream's length is unknown until read, `accept` is checked first from the part's declared type (or a 512-byte sniff), while `max_size` is enforced mid-stream — an over-limit upload is cut off `413` and the partial object is removed by the same orphan cleanup that covers every failed upload. `FileStorage.Store` receives `meta.Size` `0` and must read the reader to EOF (`storage/s3`'s uploader switches to a multipart upload; `LocalStorage` copies as-is). `upload:stream` requires `file`, is mutually exclusive with `upload:presigned`, and is rejected on a `FileKeys` list — each a registration error rather than a silent no-op. For the very largest files prefer `upload:presigned`, which never routes the bytes through the app at all.

## v0.2.3 (2026-07-18)

- **Docs:** a full documentation-correctness sweep against the source (~90 fixes across ~30 pages). Systemic drift corrected: **error codes** — `VALIDATION_FAILED`→`VALIDATION_ERROR`, `DATABASE_ERROR`→`DB_ERROR`, `METHOD_BLOCKED`→`METHOD_NOT_ALLOWED`, oversize bodies as `413 BODY_TOO_LARGE` (not `BODY_READ_ERROR`), uploads as `413 FILE_TOO_LARGE`/`415 FILE_TYPE_NOT_ALLOWED`, `RequireRole` anonymous as `403` (one instance was a failing test assertion in the tutorial); the **v0.1.3 `FilesConfig` refactor** — docs still cited the removed flat `Config.FileStorage`/`FileSignedURLTTL`/`FileMiddleware` fields (now `Config.FilesConfig.*`); the **events-vs-service split** — `service.Emit`/`Webhook`/`SendEmail` do not exist, they are `events.Emit` (DB-step middleware) and `events.Webhook`/`SendEmail` (event-bus subscribers, not middleware); and **phantom APIs** that read as real features — the SQL-backed "query model" (`QueryModelSpec`/`ModelConfig.QueryModel`), OpenAPI `OASContact`/`OASLicense`, `server.ShutdownContext`, `ActionContext`, `redis.NewCacheStore`. Non-compiling examples were fixed (`redis.New` prefix arg, `bus.Subscribe(events.Subscription{…})`, `cron.New/Add/Start`, `jobssql.Migrate` driver arg + `*sql.DB`, `service.HashField` `Hasher` arg, `validate.UniqueField` driver arg, `db.RateLimit` `KeyFunc`); wrong claims corrected (`body.MaxBodySize` is the JSON limit and also the multipart ceiling of 32 MB, so a value between the two lowers multipart; `CoerceTypes` does no `time.Time` coercion; encryption/audit fields are excluded only via `WithExcludeFields`; versioning uses a singular `{model}_history` table/route with a UNIQUE index; Postgres `40001` is not normalised to `409`; `CacheStore` is a three-method interface with no built-in Redis implementation); catalogue omissions documented (`idempotency`, `RequireLocale`, `RateLimitField`/`RateLimitAction`/`AuditLogAction`, `CopyField`/`Timestamp`, OpenAPI `SetVersion`/`AddTag`/`InjectRequestExample`); and wrong labels fixed (`middleware/db/redis` is the rate-limit `Backend` not a cache, Kafka is `segmentio/kafka-go`, RabbitMQ is `rabbitmq/amqp091-go`, catalogue packages are not each their own Go module).
- **Bugfix:** `idempotency.Middleware` now works inside an `ActionConfig.Middleware` list. An action's pipeline skips the Deserialize step that populates `ctx.RawBody`, so the middleware hashed an empty body — the "same key, different body → `422`" guard never fired, and a client could reuse an `Idempotency-Key` with a different payload and silently replay the first response, the opposite of what the key promises. The middleware now reads the body itself through the new `ctx.EnsureRawBody()`, which reads, size-limits, and buffers the body when an earlier step has not, then restores it so the action handler's `ctx.BindJSON` still sees it. Nothing changes for the Deserialize-step registration, where `ctx.RawBody` is already set, and GET/DELETE (which carry no body) are unaffected either way. `ctx.EnsureRawBody()` is exported for any middleware that needs the raw bytes before the handler binds them.
- **Security:** a `Forced` filter set before the DB step now survives the Deserialize step — which is what makes `jobs/maniflex`'s per-actor scope actually scope. `Mount` restricts a caller to their own `actor_id`/`tenant_id` job statuses by appending a filter on the **Auth** step, but Deserialize runs after Auth and rebuilt `ctx.Query` from the request wholesale, discarding it. So the scope was inert: any authenticated non-admin listed every actor's and every tenant's job statuses, and could read any status row by id (a cross-tenant leak and an existence oracle). Deserialize now carries over filters marked `FilterExpr.Forced` — a scope the server imposed — while anything derived from the request still comes from the parse alone, so a client cannot smuggle a filter in ahead of it; `jobs/maniflex` marks its filters `Forced` accordingly. Query scoping still belongs on the DB step (`db.ForceFilter`/`db.Tenancy`) for the common case, and is required when the scope must also cover writes end-to-end in one transaction; this makes a pre-DB forced filter a supported, narrower option.
- **Security:** a storage key is now bound to the principal that mints it, so a leaked key cannot be referenced onto another caller's record. Naming an existing key on a create or update checked only that the object existed, never that the caller was allowed to use it — and keys leak, through signed URLs and `file_acl:private` responses — so any authenticated caller could point their own record at another record's, or another tenant's, blob (probe-confirmed `201`), and an `auto_delete` field could then delete a blob it never owned. Every minting path — `POST /files`, a presigned upload, and a multipart upload through a model — now prefixes the key with a hash of the caller's scope (`ctx.Auth.TenantID`, else `UserID`; override via the new `FilesConfig.KeyScope`), and both reference paths — a single-key field and a `FileKeys` list — refuse a scoped key whose hash is not the referencing caller's, with `403 FILE_FORBIDDEN`. Hashing keeps the raw tenant/user id out of the key, and so out of URLs. A key minted with no principal present, or before this release, carries no marker and is left to the existence check, so upgrading breaks no reference to an already-stored key; the guarantee holds for every key minted under a principal.
- **Feature:** new `srv.RegisterRollup(maniflex.Rollup{Parent: "Order", ParentField: "paid_amount", Op: AggSum, Child: "OrderPayment", ChildField: "amount", On: "order_id"})` maintains a denormalised aggregate column on a parent from its children — the column an app otherwise recomputes by hand in every write path and which drifts the moment one path forgets. Where `ctx.Aggregate` computes on read, a rollup maintains on write: every create/update/delete of a child recomputes the affected parent from scratch, inside the child write's transaction. Recomputing rather than applying a delta is what makes it correct by construction — a delete, a re-parenting update (both old and new parent recompute), and a soft-deleted child (excluded) all fall out without a special case, and the total can never diverge from the rows it summarises. A child write with no active transaction is refused with `500 ROLLUP_NO_TX`, following the `mfx:"lock_scope"` precedent, since a child insert committing while the parent update fails is the exact drift the rollup prevents. Field names are validated at registration — a typo is a startup error naming the field, which is the whole case for the typed config over a `mfx:"rollup:sum(...)"` tag mini-language that would fail silently. `srv.BackfillRollups(ctx)` reconciles existing rows; `AggSum`/`AggCount`/`AggAvg`/`AggMin`/`AggMax` are supported.
- **Feature:** new `maniflex.FileKeys` is an `mfx:"file"` column holding many storage keys — a gallery or attachment set, which previously had no supported shape. It stores as a JSON array (JSONB on Postgres), so ordering is preserved, and every rule a single-key field enforces now binds per key: existence, `max_size` (per file, not per array), `accept`, `file_acl` signing, `auto_delete` and hard-delete cleanup. With `auto_delete`, a PATCH deletes the keys it drops once the write commits — a set difference, so reordering deletes nothing and a kept key is never touched. New `mfx:"max_count:N"` bounds the array, defaulting to 100: each key is `Stat`'d to enforce the field's rules, so an uncapped array is one request buying N storage round-trips. Write by key reference (`POST /files` or a presigned upload); multipart is refused, since it carries one file per field and would store one key while silently dropping the rest. No attachment route is mounted for a list — it streams one object, and a list names no single one.
- **Security:** `mfx:"file"` on a field that is neither a `string` nor a `maniflex.FileKeys` is now a registration error naming the type. A bare `[]string` already failed loudly at AutoMigrate ("no SQL column mapping") — but the fix that error recommends is to wrap the field in a named `SQLTyper`, and such a field then migrated cleanly, parsed `mfx:"file"` cleanly, and was skipped by **every** file path in silence: the existence/`max_size`/`accept` check, `file_acl` signing, `auto_delete` GC and hard-delete cleanup each asserted `.(string)` and fell through with no else. So the documented way around a loud error produced a field that looked protected and enforced nothing — including the `max_size`/`accept` binding added elsewhere in this release. A malformed or misplaced `mfx:"max_count:"` is likewise rejected rather than swallowed the way `min:`/`max:` swallow theirs, since `max_count:1O` would silently widen the cap from 10 to the default 100.
- **Feature:** new `validate.FieldRole("subscription_expires_at", "superuser")` gates a field on who is writing it — the write-side twin of `response.RedactField`. The `mfx:` write controls are static: `readonly`/`immutable`/`hidden` apply to every caller identically, so "only a superuser may set this column, while the owner writes the rest of their own row" could not be expressed at all, and each privileged field cost a separate endpoint. `validate.RestrictField(field, func(ctx) bool)` takes a predicate instead, for gates roles cannot express (ownership, tenant, plan tier); `FieldRole` is that with a `ctx.HasRole` check, and with no roles passed it refuses every write of the field rather than gating nothing. A caller without permission gets `403 FIELD_FORBIDDEN` rather than having the field silently dropped the way `readonly` drops one — a client sending a gated field is making a privilege error, not a schema mistake, and a `200` echoing the unchanged value back is indistinguishable from success. The write is refused whole, so a mixed PATCH changes neither field. Only a field present in the body is gated; a gate naming a field the model does not have warns once per model, since a typo would otherwise leave the real field ungated in silence.
- **Feature (breaking):** new `AddBatchComputedField` resolves a computed field for a whole page in one call, so a derived column costs one query rather than one per row — what a generated `GET /store-sites` needs to return `item_count` without a hand-written action. The callback returns one value per row, positionally; a length mismatch omits the field rather than landing values on the wrong records. One registration serves every read path: a single read and the create/update echo pass a one-row slice, and an export resolves a chunk at a time so its lazy row iterator still holds only a chunk. Per-row resolvers are now capped at 8 concurrent rather than one goroutine per row — a 100-row page fired 100 concurrent round-trips, scaling as page-size × concurrent-requests. That bounds the fan-out but does not stop a per-row resolver being an N+1; use the batch form for anything that queries. **Migration:** `ComputedFunc` now takes `*ServerContext` instead of `context.Context`, so a resolver can finally reach `ctx.Tx`, `ctx.GetModel` and `ctx.Auth` — rewrite `func(ctx context.Context, row map[string]any)` as `func(ctx *maniflex.ServerContext, row map[string]any)` and use `ctx.Ctx` where you passed the context on. The typed `AddComputedField[T]` changes the same way.
- **Feature:** computed fields now appear in the generated OpenAPI spec, as read-only properties of the model's response schema. `openapi_gen.go` never read `ModelMeta.Computed`, so a field that was returned on every read path was absent from the spec entirely and a client generated from it had no idea the field existed. They are emitted on the response schema alone, never on a create or update body, since a computed field is not writable. The new `maniflex.ComputedSchema(&maniflex.OASSchema{Type: "integer"})` declares the value's type at registration; without it the field is still emitted but carries no type, because both callbacks return `any` and inventing a type would put a claim in the spec that nothing enforces — present-but-untyped is the honest answer, and still better than absent.
- **Feature:** new `workflow.OnTransition("pending", "confirmed", applyStoreCredit)` attaches a side effect to a declared transition, running it inside the write's transaction — so it commits with the transition or not at all, and a returned error rolls the transition back. Previously the side effect went in a separate middleware that re-read the record and re-derived `from`→`to`, duplicating the state machine in a second place that could drift from it. Every matching hook fires, in declaration order, unlike `Allow`'s first-match-wins — `("*", "delivered")` and `("confirmed", "delivered")` are independent effects. Hooks need the write's transaction, which does not exist on the Validate step, so a machine that declares one registers `sm.Hooks()` on the DB step instead of `sm.Middleware()` on Validate; declaring a hook makes `Middleware()` panic rather than leave the hooks silently unfired. `Hooks()` is a strict superset — it re-reads `from` under `ctx.LockForUpdate` inside the transaction and re-checks the rule and its guards against it, which is what stops two concurrent PATCHes both reading `pending` and both applying the credit. Without an active transaction it refuses with `500 WORKFLOW_NO_TX` rather than take no lock. Guard-only machines are unaffected: `sm.Middleware()` on Validate still needs no transaction.
- **Feature (breaking):** new `mfx:"file,upload:presigned"` mounts `POST /{model}/{field}/upload-url`, which returns a presigned S3 POST policy plus the storage key: the client uploads straight to the bucket and sends the key back in the ordinary create/update, so a large file's bytes never enter the app process (a multipart upload buffers the whole body — `MaxUploadMemory` defaults to the same 32 MB as `MaxUploadBytes`). That write _is_ the completion step, so there is no pending-upload state and no call a client can skip. The route carries no record id, so a create-time file field can use it. The field's `max_size`/`accept` are checked before minting, pinned into the signature (`content-length-range`, which a presigned PUT could not do), and re-checked against the stored object on completion; the key is minted server-side via `FilesConfig.KeyGen`, never chosen by the client. Minting runs Auth under the new `maniflex.OpPresignUpload` (not `OpCreate`). **Migration:** `FileStorage` gains `Stat(ctx, key) (FileMeta, error)` and `PresignUpload(ctx, key, opts)` — third-party backends must implement both. One that cannot presign returns the new `maniflex.ErrPresignUnsupported` (→ `501`), never an unsigned URL, which would be an open write endpoint rather than a degraded presigned one. `storage/s3` does both; `LocalStorage` implements `Stat` and refuses to presign.
- **Security:** a `file` field's `max_size` and `accept` now bind when a record references a pre-uploaded storage key, not only when the bytes arrive as multipart. That path checked existence alone, so the same file the multipart path answered `413` for was accepted with a `201` when uploaded via `POST /files` and then referenced by key. Both paths now run one rule against the stored object, read via the new `FileStorage.Stat` (`Exists` answers only yes/no, and `Retrieve` would download the object to measure it). An object whose backend recorded no content type is refused when the field constrains the type, so an `accept` rule that cannot be evaluated is never quietly waived. **Migration:** a client referencing keys that violate the field's tags now gets `413 FILE_TOO_LARGE` or `415 FILE_TYPE_NOT_ALLOWED`.
- **Feature:** a `ModelConfig.Singleton` scoped by a forced filter (`db.Tenancy`/`db.ForceFilter` on the DB step) is now **one row per tenant**, resolved and provisioned per caller, instead of one row globally — the per-tenant storefront/profile/settings record the shape never fitted. There is deliberately no `SingletonScope` setting: a bare column name cannot say where the value comes from, the scoping middleware already answers that, so the scope is read from the request's forced filters. The route shape is untouched, and a request with no forced filter still gets the global `SingletonID` row, so existing apps are unaffected. This also retires a bug: `db.Tenancy` on a `Singleton` used to `404` **forever**, because provisioning inserted a filterless row that the read then filtered out. A scoped row keeps an ordinary generated primary key; give the scope column a unique index so two concurrent first accesses cannot both provision one. The alternative was `Headless` plus a hand-written action, which skips the Validate step and so gives up every `mfx` rule. **Upgrading:** an existing global row carries no scope value and matches nobody — backfill or drop it.
- **Feature:** new `Server.Execute(ctx, maniflex.Invocation{…})` runs a registered model's operation through the full pipeline from Go — the other half of `server.Action`. HTTP was the only entry point, so in-process callers had to make a request to themselves and name the principal in a header, which any client can also send; and N replayed items could not share a transaction, so a batch failing at item 3 left 1 and 2 committed. `Invocation.Auth` is a typed `*AuthInfo` and `Invocation.Tx` is the caller's transaction. Every step runs — there is no `Steps` bitmask, since skipping Auth would remove the isolation the injected principal is meant to be constrained by. `Execute` synthesises the request the pipeline already reads, so body binding (and PATCH presence semantics) and `?filter=` are the HTTP path's rather than a second copy that drifts. A non-2xx comes back as an error (`*maniflex.ExecuteError`, carrying the response), so the natural loop rolls the batch back instead of committing the prefix. `OpExport`/`OpReadAttachment` (they stream bytes) and `OpAction`/`OpSearch` (trimmed pipelines) are refused.
- **Feature:** new `ctx.InProcess()` reports whether a request came from `Server.Execute` rather than a client, and `auth.JWTAuth`/`auth.JWKSAuth` now stand aside for an in-process request that already carries a principal — an in-process call has no `Authorization` header, so the Auth step would otherwise `401` every invocation. Both halves of that condition matter: short-circuiting on `ctx.Auth != nil` alone would let an app whose earlier Auth middleware trusts a header start honouring it (measured: `X-User-Id: victim` → `200` instead of `401`). `InProcess()` reads an unexported field only `Execute` can set, so over HTTP the JWT path is unchanged and no client can claim it. An `Execute` with no principal is anonymous and still refused — being internal is not being authenticated. Use it in your own middleware for checks that are about the transport rather than the caller (CSRF, per-IP rate limits); it is also how you exempt a middleware from in-process calls, in preference to a skip-list on `Invocation`.
- **Feature:** new `db.ForceFilterVia("item", "owner_id", fn)` scopes a model through the column its `BelongsTo` parent carries — for child tables that hold only a foreign key, where `db.ForceFilter` needs a column the model does not have. The predicate engine already did this (a nested `FilterExpr` joins, and the write path's pre-flight passes the same filters to `FindByID`), so only the constructor was missing. Reads join the parent; updates and deletes read the row back through it and answer `404` on a miss. **Creates are scoped too, and had to be:** the foreign key is what the scope hangs from while being the one part of it the client supplies, so without a check `POST {"item_id": "<another tenant's>"}` landed a row under their parent, and a `PATCH` rewriting the key moved the row after it had been checked where it used to be. The named parent is now read through the scope inside the write's own transaction; a create naming no parent is `422`. There is no `TenancyVia` — the premise is a model with no column to stamp. New `ctx.ViaFilter` exposes the resolver. Registers on the DB step, so it does not cover Actions.
- **Security:** a `FilterExpr` whose operator no adapter implements is now refused rather than matching every row. The query builder fell through to `1=1`, which does not narrow a query but deletes the condition — so a forced filter with a misspelt operator was not a narrower scope but no scope, on reads and (since the write path reads back through the same filters) on writes. Only a hand-built filter could reach it, but `Operator` is a bare string type and forced filters are built in Go, not parsed: `Operator: "equals"` (the constant `OpEq` is `"eq"`) compiled, booted and served, and a cross-tenant read _and_ `PATCH` both answered `200` with nothing logged. The DB step now rejects such a filter by name, and the adapters render an unrecognised operator — and a `between` missing a bound — as `1=0`, so a filter reaching them by a path that check does not cover matches nothing rather than everything. New `maniflex.FilterOperator.Valid()`.
- **Feature:** new `db.TenancyAction` and `db.ForceFilterAction` scope a custom `Action`'s database access. `db.Tenancy`/`db.ForceFilter` register on the DB step, which an Action's pipeline does not run, so they did nothing at all for one — and nothing warned, since the ineffective-registration scan only fires when a middleware's operation filter names _only_ skipped steps. The new variants go in the action's own `Middleware` list and set a `maniflex.ActionScope` that every DB path the handler can reach consults: `ctx.GetModel`, the typed generics, `ctx.Aggregate`, `ctx.LockForUpdate` and the `Tx` from `ctx.BeginTx` apply it (so `WithTransaction` and `Batch` work unchanged), while `ctx.RawQuery`, `ctx.RawExec`, `ctx.Search` and `ctx.RecursiveQuery` **refuse** — raw SQL cannot be scoped, and scoping the convenient paths while those leaked in silence would put the guarantee in the docs and not the code. New `ctx.Unscoped()` is the deliberate, greppable bypass. Actions with no scope, and the generated CRUD routes, are unaffected.
- **Security:** `db.Tenancy` and `db.ForceFilter` now scope updates and deletes, not just reads. Their filter went into `ctx.Query`, which the write path had no way to read — the adapter's `Update`/`Delete` carry no filter where `FindByID` does — so the emitted SQL was `DELETE FROM t WHERE id = ?`, and any authenticated caller could update or delete any other tenant's row given its id, while their own reads of it correctly `404`'d. With `Tenancy` it was worse than a leak: it stamps the tenant column onto an update, so a cross-tenant `PATCH` **reassigned the row to the caller** and the owner then permanently `404`'d on their own record. The DB step now reads the record back through the request's forced filters before the write — in one transaction with it, so the row cannot leave scope in between — and answers `404` on a miss. `FilterExpr` gains `Forced` (set by `Tenancy`/`ForceFilter`, and worth setting on a hand-built filter that expresses who may touch a row), so a client's own `?filter=` still never constrains a write and an unscoped write pays for no extra read. The adapter interface is unchanged.
- **Security (breaking):** an unrecognised `mfx:` option is now a registration error naming the option it thinks you meant, instead of being discarded in silence. The list was matched exactly and case-sensitively and anything else was dropped with no error — merely puzzling for a descriptive directive, but a hole for a protective one: `Role string mfx:"read_only"` left `Readonly` false, so the field stayed writable and `PATCH {"role":"admin"}` landed. All three shapes are caught: a stray separator (`read_only`), wrong case (`Readonly`) and a transposition (`reaodnly`) all suggest `readonly`, and a value-carrying option written without its value (`lock_scope`) suggests `lock_scope:`. Where two real options are equally close the suggestion is withheld rather than guessed — `mix:` is one edit from both `min:` and `max:`. **Migration:** a model carrying an unknown option now fails to register; correct it or delete it — but check what it was _meant_ to do first, since a protective directive that never applied means the field has been unprotected all along.
- **Security:** `mfx:"hidden"` now means what it has always claimed: the client can neither read the field nor write it. Only the read half was enforced — a hidden field was scrubbed from responses but still accepted from a request body, so `IsAdmin bool` tagged `mfx:"hidden"` could be set to `true` by anyone willing to guess the column name, and because the field never appears in a response nothing revealed that it had. The documented tag table and the generated OpenAPI spec both already said "write: no"; the write path was the only place that disagreed. `hidden` now implies `readonly`. `mfx:"writeonly"` is unaffected, and spelling both (`mfx:"hidden,writeonly"`) still leaves the field writable, since an explicit directive outranks the implication. **Migration:** retag as `mfx:"writeonly"` if the client is genuinely meant to send it. Relatedly, `mfx:"hidden,required"` is now rejected at registration rather than failing every insert at runtime, and the error names `mfx:"writeonly,required"` as the combination that was meant.

## v0.2.2 (2026-07-16)

- **Security (breaking):** exports no longer hold two copies of the result set, and no longer run unbounded in parallel. Despite the "stream" naming, the endpoint built a whole second `[]map[string]any` of every row and handed it to the writer, so two full copies were live for the length of the write though each row is used once; rows are now serialised one at a time as the writer asks for them, and the CSV and XLSX writers provably hold nothing per row (20k rows of a narrow model: CSV 34.3 → 24.2 MB, XLSX 42.3 → 27.2 MB). That still leaves one export costing memory proportional to `rows × width`, and `MaxExportRows` bounds the row count of one export but neither the width of a row nor how many exports run at once — so a handful of concurrent exports of a wide model could exhaust the heap with every one of them individually inside its cap. New `Config.MaxConcurrentExports` bounds the product. **Behaviour change:** at most 4 exports now run at once by default; the next one is refused immediately with `503 EXPORT_BUSY` and a `Retry-After` rather than queued (queuing would hold a connection open behind work that is slow by nature). Raise it, or set a negative value to disable the limit, if you have admission control in front. The slot is taken before the pipeline, which is the only point at which the rows have not already been read — so a saturated server answers 503 ahead of Auth.
- **Performance:** a model's fields and relations are now indexed by name, so `FieldByDBName`/`FieldByJSONName`/`RelationByKey`/`RelationByModel` — which the pipeline calls several times per request, to resolve a filter, a sort, a cursor, an owner column, a body key — are constant-time rather than a scan of every field. The validate step's strip pass was quadratic on top of that: it walked the model's fields and, for each, called `DeleteField`, which scanned the fields again to find the one the loop already had; it is now linear. Lookup cost goes flat (~39ns at any width) where it was linear (~50ns over 8 fields, ~208ns over 40), so wide models gain the most — a 40-field model resolves ~5.3× faster. The index is built on first lookup and picks up appended entries, so a hand-built `ModelMeta` is indexed on the same terms as a registered one; a duplicate column name still resolves to the first field declaring it.
- **Performance:** removed four pieces of repeated work from the request path. `ctx.QueryParam` parses the query string at most once instead of rebuilding `url.Values` from the raw string on every lookup (reading three parameters parsed it three times); the parse is lazy, so a request that never asks for one still parses nothing. Replacing an `mfx:"auto_delete"` file re-read the record it was replacing once per file field — three such fields meant three identical reads of the same row to pull three values out of it; the pre-write row is now read once per request. `cast`, which runs per field per row of every read, dispatches on a type assertion rather than `reflect.TypeOf(value).Kind()` (~15% faster per cell, measured A/B in one binary), and that retires a latent panic: the old kind check was true for a _named_ string or bool type, but the assertion that followed it accepted only the exact type. The `?models=` validation set for `/search` is derived once at mount rather than rebuilt from the registry per request, and deciding whether a write is sourced from the typed record no longer builds — twice — a whole column map it only wanted the key names from.
- **Bugfix:** `ctx.Logger()` is now safe to call from any goroutine. It memoises the request logger on first call, and that write was unsynchronised — but the context outlives its request goroutine: a `ctx.GoBackground` closure captures it and logs from another goroutine while the request is still in later steps (the framework's own file-cleanup middleware does exactly that), so the write raced those reads. It is a data race, which `-race` flags and which is otherwise free to misbehave however the compiler and CPU like. The build is now exactly-once, so every caller gets the same logger and no two goroutines can be writing it. It stays lazy — a request that never logs still builds nothing.
- **Performance (breaking):** each pipeline step's middleware chain is now composed once per (model, operation) and cached when the router is built, instead of being rebuilt six times on every request — each rebuild walked every registered middleware, allocated before/after/skipped slices, and closed over a fresh chain, though the (model, operation) set is fixed by the time the router exists. Composition drops from O(registered middlewares) to a zero-allocation lookup: measured per step, 12 registered middlewares went from ~675 ns and 9 allocs to ~27 ns and 0, and that is paid six times per request. This also closes a data race the rebuild depended on: `Register` appended to a slice that live requests were reading, so a middleware added after boot could tear the slice header or silently go missing. **Breaking:** `Pipeline.<Step>.Register` must now be called before `Start()`/`Handler()` and panics afterwards — the same rule `AddService`, `Action`, `RealtimeDoc` and `EnableGlobalSearch` already follow. Registering late was never safe, and nothing could apply it consistently.
- **Performance:** the total-count query behind every offset list request is now a plain `COUNT(*)` instead of `COUNT(DISTINCT <table>.id)`. The `DISTINCT` was never needed: every join a list query can carry is 1:1 — a nested filter or sort joins a `BelongsTo` relation on its primary key (any other kind is rejected before it reaches SQL), and the SQLite full-text join matches one shadow row per base row on `rowid`, while `?include=` relations are loaded in separate queries and never joined in — so nothing fans the base table out. All it bought was a sort/hash dedup over the entire filtered set on every request, which also defeats an index-only count and is O(table). Totals are unchanged; large tables get the biggest win.

## v0.2.1 (2026-07-16)

- **Bugfix:** the `Server.Action` conflict check now covers every route a model mounts, not just the five CRUD ones. An action that shadowed `GET /{model}/export`, `GET /{model}/aggregate`, a per-field attachment path (`GET /{model}/{id}/{field}`), or a singleton's `GET`/`PATCH` used to pass registration — and because each model is mounted under its own sub-tree, the collision did not panic when the router built: it silently shadowed, so the model's endpoint quietly stopped answering with no error anywhere. Such an action is now rejected at the `Action()` call, naming the route it would shadow. The check is method-specific and position-based, matching how the router actually dispatches: `GET /{model}/{anyName}` collides with the read route whatever the parameter is called, while a method the model does not serve at that path (`POST /{model}/{id}`) is still free to take. It also drops a false positive the old check had — an action under a singleton's non-existent `/{id}` subtree is no longer rejected, since the singleton mounts nothing there.
- **Security (breaking):** static file serving is now opt-in. It used to fall back to `<cwd>/static` when `StaticDir` was unset, so any `static/` directory that happened to be in the working tree — a build cache, a checked-out asset bundle, a directory left over from something else — was published at `/static/` with directory listings on, without anyone asking for it. An empty `StaticDir` now serves nothing; set it to the directory you want served. Serving is otherwise unchanged: a lone file is served at its path, and a single-page app under a subdirectory (its own `index.html` plus nested assets) is served in full. **Migration:** if you relied on the implicit `./static`, add `StaticDir: "static"`. `StaticDisabled` still forces serving off even when `StaticDir` is set.
- **Bugfix (breaking):** `ConfigFromEnv` now returns `(Config, error)` and rejects a variable that is set but unreadable, instead of dropping it on the floor. `PORT=808O` used to leave the server listening on the default 8080, `QUERY_TIMEOUT_MS=abc` left it with no query timeout at all, and a negative `PORT` only failed much later inside `ListenAndServe` — in every case the deploy looked healthy and the only evidence was the behaviour you didn't get. The error names the variable and the value it could not read, and every bad variable is reported at once, so two typos cost one deploy to find rather than two; a port outside 1–65535 and an unrecognised boolean (`HEALTH_CHECK_DB=ture`) are now caught too. Unset variables are still left at their zero value for `ApplyDefaults`. **Migration:** `cfg, err := maniflex.ConfigFromEnv("PREFIX")`. Also fixes the docs, which listed `PATH_PREFIX` and `LOG_LEVEL` among the variables read (neither ever was) and showed the call without its prefix argument.
- **Bugfix:** `Server.Handler()` and `Server.Shutdown()` are now safe to call from any goroutine, at any point in the server's life. Both fields they lazily initialise were read and written unlocked: two concurrent `Handler()` calls each built their own router (and each re-resolved many-to-many relations back into the shared registry), and `Shutdown` saw the `nil` `http.Server` that stands for the whole boot window, concluded the server was not running and returned — while boot carried on behind it and opened the listener, so the common `go srv.Start(); …; srv.Shutdown(ctx)` left a bound port for the life of the process. A `Shutdown` racing boot now countermands it: the listener never opens, `Start` unwinds what it had brought up and returns `nil`, and `Shutdown` waits for that.
- **Bugfix:** the "must be called before `Start()`" guards on `AddService`, `Action`, `RealtimeDoc` and `EnableGlobalSearch` now test whether `Start` has been _entered_, rather than sentinels (`httpSrv != nil`, `router != nil`) that only went up once migration and every service had already come up. A call landing inside the boot window — the seconds a migration or a service dialling its backend takes, and precisely when a concurrent goroutine would make one — passed the guard and was then silently ignored: `lifecycle.start` has already ranged over the service slice, so a service added there is never started, and nothing said so. It now panics, as the guard always claimed it would.
- **Bugfix (breaking):** `GET /:model/aggregate` now takes its spec as URL-encoded JSON in the new `?aggregate=` query parameter, and no longer reads the request body. A body on a `GET` is stripped by many proxies, CDNs and load balancers, and `fetch()` cannot send one at all — so the endpoint worked in development and failed in production. **Migration:** move the JSON from the body to `?aggregate=${encodeURIComponent(JSON.stringify(spec))}`; a request that still sends a body gets `400 INVALID_AGGREGATE` naming the parameter. The spec itself, the field allow-list, the `OpList` dispatch and the `?filter=` merging are all unchanged. (The parameter is `aggregate` rather than `q` because `?q=` is full-text search, and this request runs as a list.)
- **Security:** the HTTP server now starts with defensive timeouts — `ReadHeaderTimeout` (new, default 10s) and `IdleTimeout` (new, default 120s). `net/http` sets none of its own, so a client could hold a connection open indefinitely by dribbling headers one byte at a time and never completing them; enough such connections exhausted the server's file descriptors without a single request reaching the pipeline (slowloris). New `ReadTimeout`/`WriteTimeout` are also exposed but stay unset by default: both are whole-request deadlines, so they would sever slow uploads and long-lived streams (SSE) respectively. A negative value disables any of the four.

## v0.1.5 (2026-07-13)

- **Feature:** new filter operators `contains`, `starts_with`, and `ends_with` take a **literal** value: `%` and `_` in it are escaped (and an explicit `ESCAPE` is emitted, so SQLite and Postgres agree), and all three are case-insensitive. `like`/`ilike` values are patterns — `?filter=label:like:50%` matches `500 units` too, and there was no portable way to escape a wildcard — so anything a user typed (a search box, a filename, an SKU) belongs in the new operators; `like`/`ilike` are unchanged and remain the right tool when the caller really is writing a pattern. `maniflex.LikePattern` builds the escaped pattern for custom adapters.
- **Bugfix:** `If-Match: *` now does what RFC 9110 says — it holds for any existing record ("overwrite whatever is there, but don't create it") instead of being string-compared against the ETag, which returned `412 Precondition Failed` on a resource that plainly existed, the one case the wildcard is defined to accept. A `*` on a missing record still returns `404`, and the row lock is taken either way, so the guarded write stays atomic.
- **Bugfix:** a failed boot no longer orphans `Server.Go` loops and background writes — when `ListenAndServe` failed (port in use) the services were stopped but the goroutines they were cancelled alongside were never awaited, and a failed migration or a service that refused to start returned from `Start` without so much as cancelling them (`Server.Go` spawns when you call it, not at `Start`). Every boot-failure exit now runs the same drain the graceful path runs, bounded by `ShutdownTimeout`, so a goroutine is never abandoned mid-write.
- **Bugfix:** `PanicRecoverer` no longer swallows `http.ErrAbortHandler` — a handler that panics with it is abandoning its response deliberately (`httputil.ReverseProxy` does this when an upstream dies mid-stream), and net/http's contract is to close the connection silently. The recoverer treated it as a crash: it logged a phantom panic at ERROR and appended a `500 PANIC` JSON envelope to a response that was already half-written and half-sent, so the client received a corrupted body instead of a truncated one. It is now re-panicked, as the stdlib expects.
- **Bugfix:** a client that hangs up mid-request is now recorded as `499` with no body, not `504 TIMEOUT` — a disconnect cancels the request context, which the pipeline reported as `"request exceeded the configured query timeout"` (a timeout that need not even be configured) and wrote to a socket nobody was reading, so every abandoned request counted against the server's timeout rate. A genuine server-side deadline still gives `504 TIMEOUT`; the disconnect is logged at `DEBUG` rather than as an error. New `maniflex.StatusClientClosedRequest` (nginx's non-standard 499) names the status for metrics and access logs.
- **Bugfix:** SQLite write connections now open their transactions with `BEGIN IMMEDIATE` (`_txlock=immediate`, added to the write DSN by `sqlite.Open`; a `_txlock` you set yourself is kept). `LockForUpdate`, the `If-Match` check and `mfx:"lock_scope"` are all documented to depend on it, but `Open` never set it: `BeginTx` issued a deferred `BEGIN` and the write-skew protection came only from the write pool being capped at one connection. Two read-then-write transactions now serialise on their own — the second waits at `BEGIN` and sees the first one's commit, instead of reading the stale row and then failing with `SQLITE_BUSY` on the way out.
- **Bugfix:** `AutoMigrate` now warns when a live column's type no longer matches its model field (`int` → `string` keeps the old `INTEGER`) — the diff only ever added missing columns and warned about extra ones, so a changed Go type was neither detected nor reported and the schema drifted silently until a read or write failed. The column is still never rewritten (that can lose data and locks the table): the warning names the table, column, and both types, and the conversion belongs in an explicit versioned migration. The check is spelling- and dialect-tolerant, so a correctly-migrated table stays quiet.
- **Bugfix:** re-deleting an already-soft-deleted row inside a transaction now returns `404` instead of succeeding — the transactional soft-delete omitted the pooled adapter's "not already deleted" guard, so the `UPDATE` still matched the row, re-stamped `deleted_at`/`is_deleted` (losing the original deletion time) and answered `204`. Affects any `DELETE` running under `WithTransaction` and the typed `maniflex.Delete` inside a `Batch`.
- **Bugfix:** `maniflex.TxFromContext` no longer hands back a finished transaction — `WithTransaction` cleared `ctx.Tx` on commit but left the same tx behind the context value, so a middleware resuming after its `next()` (an audit hook, a `jobs/sql` outbox enqueue) joined a committed transaction and failed with `sql.ErrTxDone`. Both handles are now cleared on commit _and_ on rollback, so such code falls back to the bare adapter.
- **Bugfix:** an `mfx:"auto_delete"` file's old blob is now deleted only after the update succeeds — previously it was removed before the DB write, so a failed update (e.g. a unique-constraint conflict) orphaned the row by deleting a file it still referenced.
- **Enhancement:** `realtime.SSEHandler` always sets header `X-Accel-Buffering: no` as a safe default for Nginx deployments
- **Bugfix:** the Response step no longer panics on a `ctx.DBResult` that a `Replace` middleware can plausibly produce — a `ListResult` carrying only `Items`/`Total` nil-dereferenced its absent `Query` or divided by a zero `Limit` (a 500 PANIC on every list request), and a non-pointer record panicked inside `reflect.Value.Elem`. A hand-built `ListResult` now has its pagination defaulted, and a `DBResult` of the wrong shape is reported as `500 INVALID_DB_RESULT` naming the type it got.
- **Bugfix:** `ctx.RawQuery`/`ctx.RawExec` no longer escape an active transaction — when the `Tx` cannot run raw SQL they silently fell through to the bare adapter (a different connection, outside the transaction), so a raw write a caller believed was atomic committed on its own and survived the rollback; they now fail with the new `maniflex.ErrRawNotSupportedInTx`. This also affected the framework's own traced transaction: turning on `Config.Trace` pushed every in-transaction raw statement onto the bare adapter, which now participates correctly.
- **Bugfix:** graceful shutdown now runs every phase on one shared budget, as documented — the HTTP drain, `Service.Stop` and the `OnShutdown` hook each minted a fresh `ShutdownTimeout`, so they were additive (a slow drain plus a hung stop plus a slow hook could take ~3× the timeout and get SIGKILLed mid-stop) and an explicit `Server.Shutdown(ctx)` deadline was ignored by the service/hook phase entirely. Fixed a related race on the same path: a `Shutdown()` during `StartWithContext` could have its teardown raced by the boot-failure branch, which handed services a fresh full-length window instead of the caller's deadline.
- **Bugfix:** `/openapi.json` now reflects `SetStorage`/`SetDB` — the generator held a pointer to the `Config` copy passed to `New`, not the one the server mutates, so with the two-step init (`New`, then `SetStorage`) the spec omitted the per-model attachment routes and multipart upload bodies that the router had actually mounted.
- **Bugfix:** `HEAD` is now the `GET` it mirrors with the body suppressed — it runs the same read pipeline, so it returns `404` for a record that doesn't exist (it always answered `200`) and read-scoped middleware (auth, tenancy, soft-delete) now applies, closing an existence oracle that walked around `ForOperation(OpRead)` guards. `OPTIONS` returns `204` with an `Allow` header, and neither logs the spurious "Response step reached with ctx.DBResult == nil" warning that filled logs at probe frequency. **Behaviour change:** `ctx.Operation` for a HEAD request is now `OpRead`/`OpList`; `maniflex.OpHead` is retained but never set — scope HEAD-aware middleware with `OpRead`/`OpList`.
- **Bugfix:** a nullable keyset cursor field (e.g. `PublishedAt *time.Time` as `cursor_field`) is now rejected at registration instead of silently breaking pagination — the boundary predicate is never true for NULL and the drivers disagree on where NULLs sort (Postgres last on ASC, SQLite first), so rows with a NULL cursor value were dropped or repeated across pages. **Behaviour change:** a model whose `cursor_field` is a pointer field now fails to register; make the column non-nullable or pick another cursor field.
- **Bugfix:** an `in`/`not_in` filter with no values (`?filter=role:in:`) is now rejected with `400 INVALID_QUERY` instead of emitting `role IN ()` — a syntax error that any client could use to provoke a 500 on Postgres (SQLite quietly tolerates it). A hand-built `FilterExpr` with an empty value no longer produces broken SQL either: the adapter degrades an empty `IN` to "matches nothing" and an empty `NOT IN` to "matches everything".
- **Bugfix:** an uploaded file's content type is now taken from the part's declared `Content-Type`, falling back to sniffing only when it is absent or `application/octet-stream` — sniffing previously overrode the declared type, and since it cannot recognise JSON, CSV, or office formats (they sniff as `text/plain`), a correctly-declared upload to an `mfx:"accept:application/json"` field was rejected with 415 every time.
- **Security:** multipart request bodies are now bounded — previously only the in-memory buffer was capped and the overflow spooled to temp files unchecked, so a single long upload could fill the disk. A multipart request over `FilesConfig.MaxUploadBytes` (new, default 32 MB) is rejected with `413 BODY_TOO_LARGE` as it streams, on both model uploads and `POST /files`; `FilesConfig.MaxUploadMemory` (new, default 32 MB) tunes the in-memory buffer, and a per-model `body.MaxBodySize` overrides the ceiling. **Behaviour change:** uploads larger than 32 MB now need an explicit `MaxUploadBytes`.
- **Bugfix:** an oversized JSON body is now rejected with `413 BODY_TOO_LARGE` instead of being silently truncated to the 4 MB limit — a padded body whose first 4 MB parsed as complete JSON was previously accepted, creating a record from a body the client never sent. `body.MaxBodySize` now genuinely moves the ceiling (a raised limit was still cut at 4 MB) and reports the 413 it documents; `ctx.BindJSON`/`TryBindJSON` return 413 rather than 400 for the same condition. New `ctx.SetMaxBodySize(limit)` sets the per-request ceiling.
- **Bugfix:** the `mfx:"lock_when"` guard now fails closed — a read error other than "record not found" rejects the request (500 `DB_ERROR`) instead of silently letting the update/delete through, and the guard reads through the request's active transaction so it sees uncommitted state (previously it also deadlocked against the request's own transaction on SQLite).
- **Bugfix:** `ModelConfig.OptimisticLock` no longer permits a lost update — the `If-Match` check and the write it guards now run as one row-locked transaction, so of two concurrent writers holding the same ETag exactly one succeeds and the other gets its 412 (previously both passed the check and the second silently clobbered the first).

- **Security (breaking):** `auth.JWTAuth` now rejects tokens with no `exp` claim (`401 TOKEN_MISSING_EXPIRY`, opt out via `JWTOptions.AllowNoExpiry`) and refuses an empty HMAC secret (panics; warns when the secret is under 32 bytes).
- **Security:** `maniflex.RandomString` now draws from `crypto/rand` (unbiased), making it safe for tokens/IDs; it previously used non-cryptographic `math/rand/v2`.
- **Security:** the admin CSRF cookie is now marked `Secure` when the panel is served over TLS (directly or behind a proxy setting `X-Forwarded-Proto: https`), so it can't ride a plaintext HTTP connection.
- **Feature:** fixed non-scalar types not included in OpenAPI schemas - untyped maps are added as `"type": "object"`, and added `maniflex.ObjectWithSchema` interface with `Schema() *OASSchema` for automatic schema generation for custom types in models.
- **Security:** `encryption.EnvKeyProvider` now binds the envelope version and keyID as AES-GCM additional authenticated data (new `v2` envelopes), so stored routing metadata can't be tampered with; existing `v1` envelopes still decrypt and are upgraded on key rotation.

## v0.1.3 (2026-07-06)

- `events.DeliverWithRetry` no longer swallows handler failures silently: it logs a WARN on each retried attempt and an ERROR when all attempts are exhausted (dead-lettering when a DLQ is configured, or "event dropped" when it isn't). Previously a handler that failed every time produced no log at all, so failing events vanished without a trace.
- `maniflex.List[T]` no longer returns zero rows when passed a non-nil `*QueryParams` whose `Limit` is left at its zero value. A hand-built `&QueryParams{Filters: …}` previously issued `LIMIT 0`; `Limit <= 0` (and `Page <= 0`) now fall back to the default, matching the `q == nil` case. Only the programmatic typed `List` was affected — the HTTP list path fills the limit from `?limit`.
- `GET /health` with `HealthCheckDB: true` now reports `db:"ok"` (or `"error"`) even before any model is registered. The ping set previously came only from registered models, so a bootstrapping server with a reachable `Config.DB` but no models yet reported `db:"unknown"`. `Config.DB` is now always included.
- Raw SQL (`ctx.RawQuery`/`ctx.RawExec` and the adapter's `Raw`) now rebinds `?` placeholders to the adapter's dialect, so `?` works on Postgres (`$1, $2, …`) as well as SQLite. `?` inside string literals is left untouched.
- `ctx.RawQuery` now returns the rows from a data-modifying statement with a `RETURNING` clause (`UPDATE … RETURNING`, `INSERT … RETURNING`), on both the autocommit and in-transaction paths. Previously query-vs-exec was decided by a `SELECT`-prefix check, so a `RETURNING` statement was routed to `ExecContext` and its result set was silently discarded.
- `mfx:"unique"` is now honoured when the column is added to an existing table via `ALTER TABLE ADD COLUMN`, not only when the table is first created. AutoMigrate creates a `UNIQUE INDEX` (`uidx_<table>_<column>`) on both paths. Previously the constraint was silently skipped (warn only) on the add-column path, so a uniqueness guarantee could vanish on a redeploy that added the column. **Behaviour change:** if existing rows already violate the new constraint, migration now fails with an error naming the table and column instead of skipping — resolve duplicate data before deploying.
- Added `auth.JWKSAuth(jwksURL string, opts ...JWTOptions)` for verifying asymmetric JWTs against a rotating JWK Set (an IdP's `/.well-known/jwks.json`). Keys are fetched, cached (1h TTL), and selected by the token's `kid`; an unknown `kid` triggers a rate-limited refetch so key rotation needs no redeploy. Supports RSA (`RS256/384/512`) and EC (`ES256/384/512`), stdlib only. The existing static-key `auth.JWTAuth` is unchanged.
- `jobs/sql`: added `WithTableName` (run a queue on a custom table so two queues can share a DB — the table and its indexes are renamed together) and `WithPayloadCipher` (encrypt the payload column at rest; stored values are prefixed `encq:` and decrypted on dequeue). Both are variadic options on `New`/`Migrate`, so existing calls are unchanged.
- `jobs`: a worker that has no handler for a dequeued job's type now requeues it (`Nack`) so another worker can claim it, instead of dead-lettering it. This makes a type-restricted worker safe to run against a table shared with other job types.
- Added `db.RateLimitAction` and `db.AuditLogAction` for custom `server.Action` handlers. The standard `db.RateLimit`/`db.AuditLog` register on the DB step, which actions skip — so an all-action service got no rate limiting and no audit records. The `*Action` variants run in the action's own `Middleware` list (after Auth): `RateLimitAction` keys on the caller plus request method+path; `AuditLogAction` records actor/resource/result from the action context. The actions docs now call out that DB-step middlewares don't apply to actions.
- `mfx:"encrypted"` fields are now encrypted/decrypted on the typed access paths (`maniflex.Create`/`Read`/`Update`/`List`) and `ctx.GetModel(name)`, not just the HTTP pipeline. Previously typed/background access went straight to the adapter and stored plaintext at rest (and returned undecrypted ciphertext) — the opposite of the intent for a PII column. Background contexts built with `NewBackground` should call `ctx.SetKeyProvider(srv.KeyProvider())`; a new `Server.KeyProvider()` getter exposes it. The raw adapter stays intentionally encryption-agnostic (storing ciphertext verbatim), which is what `RotateEncryptionKey` depends on.
- **Breaking:** relations are no longer inferred from a field's `ID` suffix. A `<Name>ID` field is now a plain scalar column unless you opt in with `mfx:"relation"` (target inferred by stripping `ID`, e.g. `AuthorID` → `Author`) or `mfx:"relation:Target"`. This removes the footgun where a value column that merely ended in `ID` (e.g. `CloudEventID`, `ExternalID`) silently became a `BelongsTo`. **Migration:** tag every `<Name>ID` field you want as a relation with `mfx:"relation"`; the `mfx:"norelation"` opt-out is now a deprecated no-op (still parsed so existing models compile). A bare `mfx:"relation"` on a field that doesn't end in `ID` warns at startup (a future strict mode will reject it — use `mfx:"relation:Target"`).
- **Behaviour change:** `json:"-"` now hides a field from API responses while keeping it as a real, persisted column (marked read-only so clients can't write it), instead of dropping the column entirely. This matches the Go convention that `json:"-"` affects serialization, not schema. Use `db:"-"` or `mfx:"-"` to exclude a field from persistence. Models that used `json:"-"` to omit a column will now get one on the next migration (AutoMigrate only adds columns, never drops).
- **Breaking:** replaced `Config.AutoMigrate bool` with `Config.DisableAutoMigrate bool`. Migration now genuinely runs by default (the old `AutoMigrate` documented "Default: true" but its zero value was `false`, so a server built from a bare `Config` or `ConfigFromEnv` silently skipped migration and 500'd on the first query). **Migration:** delete `AutoMigrate: true` (now the default); replace `AutoMigrate: false` with `DisableAutoMigrate: true`.
- **Breaking:** the file-upload settings on `Config` are consolidated into a single `Config.FilesConfig` struct. The flat `FileStorage`, `FileSignedURLTTL`, and `FileMiddleware` fields are removed and become `FilesConfig.Storage`, `FilesConfig.SignedURLTTL`, and `FilesConfig.BeforeMiddlewares`. The standalone `/files` endpoints (`POST /files`, `GET /files/*`, `DELETE /files/*`) are also no longer auto-mounted just by configuring a storage backend — you must opt in with `FilesConfig.MountEndpoints: true`. Per-model `mfx:"file"` attachment routes are unaffected: they mount off `FilesConfig.Storage` alone and still run the full Auth/DB pipeline. Two new hooks land alongside: `FilesConfig.KeyGen` overrides the storage key derived for a standalone upload (default `DefaultKeyGen` → `uploads/<uuid>/<sanitised-filename>`, now exported, you can use it for per-tenant prefixes read from `ctx.Auth` for example), and `FilesConfig.AfterMiddlewares` observe the outcome of a `/files` request for audit/metrics only (the response is already streamed, so read the status via `ctx.Writer.(interface{ Status() int }).Status()`; to alter or block a request use `BeforeMiddlewares`). Mounting the endpoints while `Storage` is still nil is allowed and returns `501 NO_STORAGE` per request. **Migration:** move `FileStorage: fs` into `FilesConfig: FilesConfig{Storage: fs, MountEndpoints: true}` (the `MountEndpoints: true` is required to keep `/files` reachable — previously a non-nil `FileStorage` mounted them implicitly); rename `FileSignedURLTTL` → `FilesConfig.SignedURLTTL` and `FileMiddleware` → `FilesConfig.BeforeMiddlewares`. `Server.SetStorage` is unchanged (it still sets `FilesConfig.Storage`).

## v0.1.2 (2026-06-20)

- Docs: corrected the `postgres.Open` examples to the real positional signature `Open(writeDSN, readDSN, registry)` (pool/session tuning via `OpenWithConfig`); documented custom-action behaviours (empty `ctx.Files`, the SQLite single-writer transaction caveat, streaming raw bytes via `ctx.Writer`, emitting events from an action), the `mfx:"file"` pre-uploaded-key existence check, and searching localized (`LocaleString`) content.
- Added `ModelConfig.Headless`: registers a model fully (migration, registry, typed access, relations) but mounts no REST routes, freeing its table path for a custom `server.Action` without the chi route-collision panic. Headless models are also excluded from the generated OpenAPI paths (their schema is still emitted).
- Convention foreign keys (`<Name>ID` fields auto-promoted to a `BelongsTo`) now log a warning at startup when the target model was never registered — the common case of storing a foreign id by design in a microservice. Tag the field `mfx:"norelation"` to silence it. The relation is still created; this only warns.
- Omitting a JSON / `SQLTyper` column (`LocaleString`, JSON maps/arrays, `money.Amount`, …) on create no longer fails with a `NOT NULL` violation: the migrator now gives such columns a synthesised empty-container default (`{}` / `[]` / the type's `driver.Valuer` zero, e.g. `0.0000`). Set `mfx:"required"` to opt a column out of the default.
- Database `NOT NULL` violations now return `422 VALIDATION_ERROR` (missing required field) instead of an opaque `500` or a misleading `409`. `maniflex.ErrConstraint` gained a `Kind` field (`unique` / `foreign_key` / `not_null`) and the SQLite/Postgres error normalisers classify accordingly.
- Removed the documented-but-nonexistent `auth.AllowPublicWrite()` from the docs. Public sign-up is achieved by scoping `JWTAuth` away from the operation (auth scoping is inclusion-only), which the auth tutorial, FAQ, and reference now show.
- Create and update responses now serialise JSON / `SQLTyper` columns (`LocaleString`, JSON-backed `sql.Scanner` types) as objects/arrays, matching the read/list responses, instead of returning them as raw JSON strings. Previously the create/update echo went through the map write path, which carried these columns as the stored text; reads already returned structured values, so the two disagreed.
- Fixed `Register`/`MustRegister` panicking ("model must embed BaseModel") when a `ModelConfig` is inlined inside a slice argument. The `ModelConfig`-binds-to-the-preceding-model rule now applies inside flattened slices too, so the documented `MustRegister(domainA.Models(), domainB.Models())` layout — where each `Models() []any` carries its own inline `ModelConfig`s — works as intended.
- Fixed a silent OR-vs-AND footgun in programmatically built filters: a `maniflex.FilterExpr` left at its zero value (`Group == 0`) is now treated as ungrouped and AND-ed, instead of being OR-ed into "group 0". Multiple hand-built filters (e.g. an ownership-scoped `user_id` + `archived` list) now AND together as intended rather than silently matching extra rows. Internally, OR groups are `Group >= 1` and the URL `?filter[N]=` syntax maps onto `Group N+1`, so the HTTP contract (`?filter[0]=a&filter[0]=b` = OR) is unchanged. Code that built OR groups by hand using `Group: 0` must switch to a positive group index.
- `AutoMigrate` now fails fast with a clear error when a model declares a field whose Go type has no SQL column mapping (e.g. a bare `map[string]any` or `[]string`). Previously such columns were silently dropped, so the model registered and "migrated" successfully while the field was never persisted. Wrap the field in a named type implementing `maniflex.SQLTyper`, or exclude it with the `mfx:"-"` tag.
