# Changelog

## v0.2.3

- **Feature (breaking):** new `mfx:"file,upload:presigned"` mounts `POST /{model}/{field}/upload-url`, which returns a presigned S3 POST policy plus the storage key: the client uploads straight to the bucket and sends the key back in the ordinary create/update, so a large file's bytes never enter the app process (a multipart upload buffers the whole body ‚Äî `MaxUploadMemory` defaults to the same 32 MB as `MaxUploadBytes`). That write *is* the completion step, so there is no pending-upload state and no call a client can skip. The route carries no record id, so a create-time file field can use it. The field's `max_size`/`accept` are checked before minting, pinned into the signature (`content-length-range`, which a presigned PUT could not do), and re-checked against the stored object on completion; the key is minted server-side via `FilesConfig.KeyGen`, never chosen by the client. Minting runs Auth under the new `maniflex.OpPresignUpload` (not `OpCreate`). **Migration:** `FileStorage` gains `Stat(ctx, key) (FileMeta, error)` and `PresignUpload(ctx, key, opts)` ‚Äî third-party backends must implement both. One that cannot presign returns the new `maniflex.ErrPresignUnsupported` (‚Üí `501`), never an unsigned URL, which would be an open write endpoint rather than a degraded presigned one. `storage/s3` does both; `LocalStorage` implements `Stat` and refuses to presign.
- **Security:** a `file` field's `max_size` and `accept` now bind when a record references a pre-uploaded storage key, not only when the bytes arrive as multipart. That path checked existence alone, so the same file the multipart path answered `413` for was accepted with a `201` when uploaded via `POST /files` and then referenced by key. Both paths now run one rule against the stored object, read via the new `FileStorage.Stat` (`Exists` answers only yes/no, and `Retrieve` would download the object to measure it). An object whose backend recorded no content type is refused when the field constrains the type, so an `accept` rule that cannot be evaluated is never quietly waived. **Migration:** a client referencing keys that violate the field's tags now gets `413 FILE_TOO_LARGE` or `415 FILE_TYPE_NOT_ALLOWED`.
- **Feature:** a `ModelConfig.Singleton` scoped by a forced filter (`db.Tenancy`/`db.ForceFilter` on the DB step) is now **one row per tenant**, resolved and provisioned per caller, instead of one row globally ‚Äî the per-tenant storefront/profile/settings record the shape never fitted. There is deliberately no `SingletonScope` setting: a bare column name cannot say where the value comes from, the scoping middleware already answers that, so the scope is read from the request's forced filters. The route shape is untouched, and a request with no forced filter still gets the global `SingletonID` row, so existing apps are unaffected. This also retires a bug: `db.Tenancy` on a `Singleton` used to `404` **forever**, because provisioning inserted a filterless row that the read then filtered out. A scoped row keeps an ordinary generated primary key; give the scope column a unique index so two concurrent first accesses cannot both provision one. The alternative was `Headless` plus a hand-written action, which skips the Validate step and so gives up every `mfx` rule. **Upgrading:** an existing global row carries no scope value and matches nobody ‚Äî backfill or drop it.
- **Feature:** new `Server.Execute(ctx, maniflex.Invocation{‚Ä¶})` runs a registered model's operation through the full pipeline from Go ‚Äî the other half of `server.Action`. HTTP was the only entry point, so in-process callers had to make a request to themselves and name the principal in a header, which any client can also send; and N replayed items could not share a transaction, so a batch failing at item 3 left 1 and 2 committed. `Invocation.Auth` is a typed `*AuthInfo` and `Invocation.Tx` is the caller's transaction. Every step runs ‚Äî there is no `Steps` bitmask, since skipping Auth would remove the isolation the injected principal is meant to be constrained by. `Execute` synthesises the request the pipeline already reads, so body binding (and PATCH presence semantics) and `?filter=` are the HTTP path's rather than a second copy that drifts. A non-2xx comes back as an error (`*maniflex.ExecuteError`, carrying the response), so the natural loop rolls the batch back instead of committing the prefix. `OpExport`/`OpReadAttachment` (they stream bytes) and `OpAction`/`OpSearch` (trimmed pipelines) are refused.
- **Feature:** new `ctx.InProcess()` reports whether a request came from `Server.Execute` rather than a client, and `auth.JWTAuth`/`auth.JWKSAuth` now stand aside for an in-process request that already carries a principal ‚Äî an in-process call has no `Authorization` header, so the Auth step would otherwise `401` every invocation. Both halves of that condition matter: short-circuiting on `ctx.Auth != nil` alone would let an app whose earlier Auth middleware trusts a header start honouring it (measured: `X-User-Id: victim` ‚Üí `200` instead of `401`). `InProcess()` reads an unexported field only `Execute` can set, so over HTTP the JWT path is unchanged and no client can claim it. An `Execute` with no principal is anonymous and still refused ‚Äî being internal is not being authenticated. Use it in your own middleware for checks that are about the transport rather than the caller (CSRF, per-IP rate limits); it is also how you exempt a middleware from in-process calls, in preference to a skip-list on `Invocation`.
- **Feature:** new `db.ForceFilterVia("item", "owner_id", fn)` scopes a model through the column its `BelongsTo` parent carries ‚Äî for child tables that hold only a foreign key, where `db.ForceFilter` needs a column the model does not have. The predicate engine already did this (a nested `FilterExpr` joins, and the write path's pre-flight passes the same filters to `FindByID`), so only the constructor was missing. Reads join the parent; updates and deletes read the row back through it and answer `404` on a miss. **Creates are scoped too, and had to be:** the foreign key is what the scope hangs from while being the one part of it the client supplies, so without a check `POST {"item_id": "<another tenant's>"}` landed a row under their parent, and a `PATCH` rewriting the key moved the row after it had been checked where it used to be. The named parent is now read through the scope inside the write's own transaction; a create naming no parent is `422`. There is no `TenancyVia` ‚Äî the premise is a model with no column to stamp. New `ctx.ViaFilter` exposes the resolver. Registers on the DB step, so it does not cover Actions.
- **Security:** a `FilterExpr` whose operator no adapter implements is now refused rather than matching every row. The query builder fell through to `1=1`, which does not narrow a query but deletes the condition ‚Äî so a forced filter with a misspelt operator was not a narrower scope but no scope, on reads and (since the write path reads back through the same filters) on writes. Only a hand-built filter could reach it, but `Operator` is a bare string type and forced filters are built in Go, not parsed: `Operator: "equals"` (the constant `OpEq` is `"eq"`) compiled, booted and served, and a cross-tenant read *and* `PATCH` both answered `200` with nothing logged. The DB step now rejects such a filter by name, and the adapters render an unrecognised operator ‚Äî and a `between` missing a bound ‚Äî as `1=0`, so a filter reaching them by a path that check does not cover matches nothing rather than everything. New `maniflex.FilterOperator.Valid()`.
- **Feature:** new `db.TenancyAction` and `db.ForceFilterAction` scope a custom `Action`'s database access. `db.Tenancy`/`db.ForceFilter` register on the DB step, which an Action's pipeline does not run, so they did nothing at all for one ‚Äî and nothing warned, since the ineffective-registration scan only fires when a middleware's operation filter names *only* skipped steps. The new variants go in the action's own `Middleware` list and set a `maniflex.ActionScope` that every DB path the handler can reach consults: `ctx.GetModel`, the typed generics, `ctx.Aggregate`, `ctx.LockForUpdate` and the `Tx` from `ctx.BeginTx` apply it (so `WithTransaction` and `Batch` work unchanged), while `ctx.RawQuery`, `ctx.RawExec`, `ctx.Search` and `ctx.RecursiveQuery` **refuse** ‚Äî raw SQL cannot be scoped, and scoping the convenient paths while those leaked in silence would put the guarantee in the docs and not the code. New `ctx.Unscoped()` is the deliberate, greppable bypass. Actions with no scope, and the generated CRUD routes, are unaffected.
- **Security:** `db.Tenancy` and `db.ForceFilter` now scope updates and deletes, not just reads. Their filter went into `ctx.Query`, which the write path had no way to read ‚Äî the adapter's `Update`/`Delete` carry no filter where `FindByID` does ‚Äî so the emitted SQL was `DELETE FROM t WHERE id = ?`, and any authenticated caller could update or delete any other tenant's row given its id, while their own reads of it correctly `404`'d. With `Tenancy` it was worse than a leak: it stamps the tenant column onto an update, so a cross-tenant `PATCH` **reassigned the row to the caller** and the owner then permanently `404`'d on their own record. The DB step now reads the record back through the request's forced filters before the write ‚Äî in one transaction with it, so the row cannot leave scope in between ‚Äî and answers `404` on a miss. `FilterExpr` gains `Forced` (set by `Tenancy`/`ForceFilter`, and worth setting on a hand-built filter that expresses who may touch a row), so a client's own `?filter=` still never constrains a write and an unscoped write pays for no extra read. The adapter interface is unchanged.
- **Security (breaking):** an unrecognised `mfx:` option is now a registration error naming the option it thinks you meant, instead of being discarded in silence. The list was matched exactly and case-sensitively and anything else was dropped with no error ‚Äî merely puzzling for a descriptive directive, but a hole for a protective one: `Role string mfx:"read_only"` left `Readonly` false, so the field stayed writable and `PATCH {"role":"admin"}` landed. All three shapes are caught: a stray separator (`read_only`), wrong case (`Readonly`) and a transposition (`reaodnly`) all suggest `readonly`, and a value-carrying option written without its value (`lock_scope`) suggests `lock_scope:`. Where two real options are equally close the suggestion is withheld rather than guessed ‚Äî `mix:` is one edit from both `min:` and `max:`. **Migration:** a model carrying an unknown option now fails to register; correct it or delete it ‚Äî but check what it was *meant* to do first, since a protective directive that never applied means the field has been unprotected all along.
- **Security:** `mfx:"hidden"` now means what it has always claimed: the client can neither read the field nor write it. Only the read half was enforced ‚Äî a hidden field was scrubbed from responses but still accepted from a request body, so `IsAdmin bool` tagged `mfx:"hidden"` could be set to `true` by anyone willing to guess the column name, and because the field never appears in a response nothing revealed that it had. The documented tag table and the generated OpenAPI spec both already said "write: no"; the write path was the only place that disagreed. `hidden` now implies `readonly`. `mfx:"writeonly"` is unaffected, and spelling both (`mfx:"hidden,writeonly"`) still leaves the field writable, since an explicit directive outranks the implication. **Migration:** retag as `mfx:"writeonly"` if the client is genuinely meant to send it. Relatedly, `mfx:"hidden,required"` is now rejected at registration rather than failing every insert at runtime, and the error names `mfx:"writeonly,required"` as the combination that was meant.

## v0.2.2 (2026-07-16)

- **Security (breaking):** exports no longer hold two copies of the result set, and no longer run unbounded in parallel. Despite the "stream" naming, the endpoint built a whole second `[]map[string]any` of every row and handed it to the writer, so two full copies were live for the length of the write though each row is used once; rows are now serialised one at a time as the writer asks for them, and the CSV and XLSX writers provably hold nothing per row (20k rows of a narrow model: CSV 34.3 ‚Üí 24.2 MB, XLSX 42.3 ‚Üí 27.2 MB). That still leaves one export costing memory proportional to `rows √ó width`, and `MaxExportRows` bounds the row count of one export but neither the width of a row nor how many exports run at once ‚Äî so a handful of concurrent exports of a wide model could exhaust the heap with every one of them individually inside its cap. New `Config.MaxConcurrentExports` bounds the product. **Behaviour change:** at most 4 exports now run at once by default; the next one is refused immediately with `503 EXPORT_BUSY` and a `Retry-After` rather than queued (queuing would hold a connection open behind work that is slow by nature). Raise it, or set a negative value to disable the limit, if you have admission control in front. The slot is taken before the pipeline, which is the only point at which the rows have not already been read ‚Äî so a saturated server answers 503 ahead of Auth.
- **Performance:** a model's fields and relations are now indexed by name, so `FieldByDBName`/`FieldByJSONName`/`RelationByKey`/`RelationByModel` ‚Äî which the pipeline calls several times per request, to resolve a filter, a sort, a cursor, an owner column, a body key ‚Äî are constant-time rather than a scan of every field. The validate step's strip pass was quadratic on top of that: it walked the model's fields and, for each, called `DeleteField`, which scanned the fields again to find the one the loop already had; it is now linear. Lookup cost goes flat (~39ns at any width) where it was linear (~50ns over 8 fields, ~208ns over 40), so wide models gain the most ‚Äî a 40-field model resolves ~5.3√ó faster. The index is built on first lookup and picks up appended entries, so a hand-built `ModelMeta` is indexed on the same terms as a registered one; a duplicate column name still resolves to the first field declaring it.
- **Performance:** removed four pieces of repeated work from the request path. `ctx.QueryParam` parses the query string at most once instead of rebuilding `url.Values` from the raw string on every lookup (reading three parameters parsed it three times); the parse is lazy, so a request that never asks for one still parses nothing. Replacing an `mfx:"auto_delete"` file re-read the record it was replacing once per file field ‚Äî three such fields meant three identical reads of the same row to pull three values out of it; the pre-write row is now read once per request. `cast`, which runs per field per row of every read, dispatches on a type assertion rather than `reflect.TypeOf(value).Kind()` (~15% faster per cell, measured A/B in one binary), and that retires a latent panic: the old kind check was true for a _named_ string or bool type, but the assertion that followed it accepted only the exact type. The `?models=` validation set for `/search` is derived once at mount rather than rebuilt from the registry per request, and deciding whether a write is sourced from the typed record no longer builds ‚Äî twice ‚Äî a whole column map it only wanted the key names from.
- **Bugfix:** `ctx.Logger()` is now safe to call from any goroutine. It memoises the request logger on first call, and that write was unsynchronised ‚Äî but the context outlives its request goroutine: a `ctx.GoBackground` closure captures it and logs from another goroutine while the request is still in later steps (the framework's own file-cleanup middleware does exactly that), so the write raced those reads. It is a data race, which `-race` flags and which is otherwise free to misbehave however the compiler and CPU like. The build is now exactly-once, so every caller gets the same logger and no two goroutines can be writing it. It stays lazy ‚Äî a request that never logs still builds nothing.
- **Performance (breaking):** each pipeline step's middleware chain is now composed once per (model, operation) and cached when the router is built, instead of being rebuilt six times on every request ‚Äî each rebuild walked every registered middleware, allocated before/after/skipped slices, and closed over a fresh chain, though the (model, operation) set is fixed by the time the router exists. Composition drops from O(registered middlewares) to a zero-allocation lookup: measured per step, 12 registered middlewares went from ~675 ns and 9 allocs to ~27 ns and 0, and that is paid six times per request. This also closes a data race the rebuild depended on: `Register` appended to a slice that live requests were reading, so a middleware added after boot could tear the slice header or silently go missing. **Breaking:** `Pipeline.<Step>.Register` must now be called before `Start()`/`Handler()` and panics afterwards ‚Äî the same rule `AddService`, `Action`, `RealtimeDoc` and `EnableGlobalSearch` already follow. Registering late was never safe, and nothing could apply it consistently.
- **Performance:** the total-count query behind every offset list request is now a plain `COUNT(*)` instead of `COUNT(DISTINCT <table>.id)`. The `DISTINCT` was never needed: every join a list query can carry is 1:1 ‚Äî a nested filter or sort joins a `BelongsTo` relation on its primary key (any other kind is rejected before it reaches SQL), and the SQLite full-text join matches one shadow row per base row on `rowid`, while `?include=` relations are loaded in separate queries and never joined in ‚Äî so nothing fans the base table out. All it bought was a sort/hash dedup over the entire filtered set on every request, which also defeats an index-only count and is O(table). Totals are unchanged; large tables get the biggest win.

## v0.2.1 (2026-07-16)

- **Bugfix:** the `Server.Action` conflict check now covers every route a model mounts, not just the five CRUD ones. An action that shadowed `GET /{model}/export`, `GET /{model}/aggregate`, a per-field attachment path (`GET /{model}/{id}/{field}`), or a singleton's `GET`/`PATCH` used to pass registration ‚Äî and because each model is mounted under its own sub-tree, the collision did not panic when the router built: it silently shadowed, so the model's endpoint quietly stopped answering with no error anywhere. Such an action is now rejected at the `Action()` call, naming the route it would shadow. The check is method-specific and position-based, matching how the router actually dispatches: `GET /{model}/{anyName}` collides with the read route whatever the parameter is called, while a method the model does not serve at that path (`POST /{model}/{id}`) is still free to take. It also drops a false positive the old check had ‚Äî an action under a singleton's non-existent `/{id}` subtree is no longer rejected, since the singleton mounts nothing there.
- **Security (breaking):** static file serving is now opt-in. It used to fall back to `<cwd>/static` when `StaticDir` was unset, so any `static/` directory that happened to be in the working tree ‚Äî a build cache, a checked-out asset bundle, a directory left over from something else ‚Äî was published at `/static/` with directory listings on, without anyone asking for it. An empty `StaticDir` now serves nothing; set it to the directory you want served. Serving is otherwise unchanged: a lone file is served at its path, and a single-page app under a subdirectory (its own `index.html` plus nested assets) is served in full. **Migration:** if you relied on the implicit `./static`, add `StaticDir: "static"`. `StaticDisabled` still forces serving off even when `StaticDir` is set.
- **Bugfix (breaking):** `ConfigFromEnv` now returns `(Config, error)` and rejects a variable that is set but unreadable, instead of dropping it on the floor. `PORT=808O` used to leave the server listening on the default 8080, `QUERY_TIMEOUT_MS=abc` left it with no query timeout at all, and a negative `PORT` only failed much later inside `ListenAndServe` ‚Äî in every case the deploy looked healthy and the only evidence was the behaviour you didn't get. The error names the variable and the value it could not read, and every bad variable is reported at once, so two typos cost one deploy to find rather than two; a port outside 1‚Äì65535 and an unrecognised boolean (`HEALTH_CHECK_DB=ture`) are now caught too. Unset variables are still left at their zero value for `ApplyDefaults`. **Migration:** `cfg, err := maniflex.ConfigFromEnv("PREFIX")`. Also fixes the docs, which listed `PATH_PREFIX` and `LOG_LEVEL` among the variables read (neither ever was) and showed the call without its prefix argument.
- **Bugfix:** `Server.Handler()` and `Server.Shutdown()` are now safe to call from any goroutine, at any point in the server's life. Both fields they lazily initialise were read and written unlocked: two concurrent `Handler()` calls each built their own router (and each re-resolved many-to-many relations back into the shared registry), and `Shutdown` saw the `nil` `http.Server` that stands for the whole boot window, concluded the server was not running and returned ‚Äî while boot carried on behind it and opened the listener, so the common `go srv.Start(); ‚Ä¶; srv.Shutdown(ctx)` left a bound port for the life of the process. A `Shutdown` racing boot now countermands it: the listener never opens, `Start` unwinds what it had brought up and returns `nil`, and `Shutdown` waits for that.
- **Bugfix:** the "must be called before `Start()`" guards on `AddService`, `Action`, `RealtimeDoc` and `EnableGlobalSearch` now test whether `Start` has been _entered_, rather than sentinels (`httpSrv != nil`, `router != nil`) that only went up once migration and every service had already come up. A call landing inside the boot window ‚Äî the seconds a migration or a service dialling its backend takes, and precisely when a concurrent goroutine would make one ‚Äî passed the guard and was then silently ignored: `lifecycle.start` has already ranged over the service slice, so a service added there is never started, and nothing said so. It now panics, as the guard always claimed it would.
- **Bugfix (breaking):** `GET /:model/aggregate` now takes its spec as URL-encoded JSON in the new `?aggregate=` query parameter, and no longer reads the request body. A body on a `GET` is stripped by many proxies, CDNs and load balancers, and `fetch()` cannot send one at all ‚Äî so the endpoint worked in development and failed in production. **Migration:** move the JSON from the body to `?aggregate=${encodeURIComponent(JSON.stringify(spec))}`; a request that still sends a body gets `400 INVALID_AGGREGATE` naming the parameter. The spec itself, the field allow-list, the `OpList` dispatch and the `?filter=` merging are all unchanged. (The parameter is `aggregate` rather than `q` because `?q=` is full-text search, and this request runs as a list.)
- **Security:** the HTTP server now starts with defensive timeouts ‚Äî `ReadHeaderTimeout` (new, default 10s) and `IdleTimeout` (new, default 120s). `net/http` sets none of its own, so a client could hold a connection open indefinitely by dribbling headers one byte at a time and never completing them; enough such connections exhausted the server's file descriptors without a single request reaching the pipeline (slowloris). New `ReadTimeout`/`WriteTimeout` are also exposed but stay unset by default: both are whole-request deadlines, so they would sever slow uploads and long-lived streams (SSE) respectively. A negative value disables any of the four.

## v0.1.5 (2026-07-13)

- **Feature:** new filter operators `contains`, `starts_with`, and `ends_with` take a **literal** value: `%` and `_` in it are escaped (and an explicit `ESCAPE` is emitted, so SQLite and Postgres agree), and all three are case-insensitive. `like`/`ilike` values are patterns ‚Äî `?filter=label:like:50%` matches `500 units` too, and there was no portable way to escape a wildcard ‚Äî so anything a user typed (a search box, a filename, an SKU) belongs in the new operators; `like`/`ilike` are unchanged and remain the right tool when the caller really is writing a pattern. `maniflex.LikePattern` builds the escaped pattern for custom adapters.
- **Bugfix:** `If-Match: *` now does what RFC 9110 says ‚Äî it holds for any existing record ("overwrite whatever is there, but don't create it") instead of being string-compared against the ETag, which returned `412 Precondition Failed` on a resource that plainly existed, the one case the wildcard is defined to accept. A `*` on a missing record still returns `404`, and the row lock is taken either way, so the guarded write stays atomic.
- **Bugfix:** a failed boot no longer orphans `Server.Go` loops and background writes ‚Äî when `ListenAndServe` failed (port in use) the services were stopped but the goroutines they were cancelled alongside were never awaited, and a failed migration or a service that refused to start returned from `Start` without so much as cancelling them (`Server.Go` spawns when you call it, not at `Start`). Every boot-failure exit now runs the same drain the graceful path runs, bounded by `ShutdownTimeout`, so a goroutine is never abandoned mid-write.
- **Bugfix:** `PanicRecoverer` no longer swallows `http.ErrAbortHandler` ‚Äî a handler that panics with it is abandoning its response deliberately (`httputil.ReverseProxy` does this when an upstream dies mid-stream), and net/http's contract is to close the connection silently. The recoverer treated it as a crash: it logged a phantom panic at ERROR and appended a `500 PANIC` JSON envelope to a response that was already half-written and half-sent, so the client received a corrupted body instead of a truncated one. It is now re-panicked, as the stdlib expects.
- **Bugfix:** a client that hangs up mid-request is now recorded as `499` with no body, not `504 TIMEOUT` ‚Äî a disconnect cancels the request context, which the pipeline reported as `"request exceeded the configured query timeout"` (a timeout that need not even be configured) and wrote to a socket nobody was reading, so every abandoned request counted against the server's timeout rate. A genuine server-side deadline still gives `504 TIMEOUT`; the disconnect is logged at `DEBUG` rather than as an error. New `maniflex.StatusClientClosedRequest` (nginx's non-standard 499) names the status for metrics and access logs.
- **Bugfix:** SQLite write connections now open their transactions with `BEGIN IMMEDIATE` (`_txlock=immediate`, added to the write DSN by `sqlite.Open`; a `_txlock` you set yourself is kept). `LockForUpdate`, the `If-Match` check and `mfx:"lock_scope"` are all documented to depend on it, but `Open` never set it: `BeginTx` issued a deferred `BEGIN` and the write-skew protection came only from the write pool being capped at one connection. Two read-then-write transactions now serialise on their own ‚Äî the second waits at `BEGIN` and sees the first one's commit, instead of reading the stale row and then failing with `SQLITE_BUSY` on the way out.
- **Bugfix:** `AutoMigrate` now warns when a live column's type no longer matches its model field (`int` ‚Üí `string` keeps the old `INTEGER`) ‚Äî the diff only ever added missing columns and warned about extra ones, so a changed Go type was neither detected nor reported and the schema drifted silently until a read or write failed. The column is still never rewritten (that can lose data and locks the table): the warning names the table, column, and both types, and the conversion belongs in an explicit versioned migration. The check is spelling- and dialect-tolerant, so a correctly-migrated table stays quiet.
- **Bugfix:** re-deleting an already-soft-deleted row inside a transaction now returns `404` instead of succeeding ‚Äî the transactional soft-delete omitted the pooled adapter's "not already deleted" guard, so the `UPDATE` still matched the row, re-stamped `deleted_at`/`is_deleted` (losing the original deletion time) and answered `204`. Affects any `DELETE` running under `WithTransaction` and the typed `maniflex.Delete` inside a `Batch`.
- **Bugfix:** `maniflex.TxFromContext` no longer hands back a finished transaction ‚Äî `WithTransaction` cleared `ctx.Tx` on commit but left the same tx behind the context value, so a middleware resuming after its `next()` (an audit hook, a `jobs/sql` outbox enqueue) joined a committed transaction and failed with `sql.ErrTxDone`. Both handles are now cleared on commit _and_ on rollback, so such code falls back to the bare adapter.
- **Bugfix:** an `mfx:"auto_delete"` file's old blob is now deleted only after the update succeeds ‚Äî previously it was removed before the DB write, so a failed update (e.g. a unique-constraint conflict) orphaned the row by deleting a file it still referenced.
- **Enhancement:** `realtime.SSEHandler` always sets header `X-Accel-Buffering: no` as a safe default for Nginx deployments
- **Bugfix:** the Response step no longer panics on a `ctx.DBResult` that a `Replace` middleware can plausibly produce ‚Äî a `ListResult` carrying only `Items`/`Total` nil-dereferenced its absent `Query` or divided by a zero `Limit` (a 500 PANIC on every list request), and a non-pointer record panicked inside `reflect.Value.Elem`. A hand-built `ListResult` now has its pagination defaulted, and a `DBResult` of the wrong shape is reported as `500 INVALID_DB_RESULT` naming the type it got.
- **Bugfix:** `ctx.RawQuery`/`ctx.RawExec` no longer escape an active transaction ‚Äî when the `Tx` cannot run raw SQL they silently fell through to the bare adapter (a different connection, outside the transaction), so a raw write a caller believed was atomic committed on its own and survived the rollback; they now fail with the new `maniflex.ErrRawNotSupportedInTx`. This also affected the framework's own traced transaction: turning on `Config.Trace` pushed every in-transaction raw statement onto the bare adapter, which now participates correctly.
- **Bugfix:** graceful shutdown now runs every phase on one shared budget, as documented ‚Äî the HTTP drain, `Service.Stop` and the `OnShutdown` hook each minted a fresh `ShutdownTimeout`, so they were additive (a slow drain plus a hung stop plus a slow hook could take ~3√ó the timeout and get SIGKILLed mid-stop) and an explicit `Server.Shutdown(ctx)` deadline was ignored by the service/hook phase entirely. Fixed a related race on the same path: a `Shutdown()` during `StartWithContext` could have its teardown raced by the boot-failure branch, which handed services a fresh full-length window instead of the caller's deadline.
- **Bugfix:** `/openapi.json` now reflects `SetStorage`/`SetDB` ‚Äî the generator held a pointer to the `Config` copy passed to `New`, not the one the server mutates, so with the two-step init (`New`, then `SetStorage`) the spec omitted the per-model attachment routes and multipart upload bodies that the router had actually mounted.
- **Bugfix:** `HEAD` is now the `GET` it mirrors with the body suppressed ‚Äî it runs the same read pipeline, so it returns `404` for a record that doesn't exist (it always answered `200`) and read-scoped middleware (auth, tenancy, soft-delete) now applies, closing an existence oracle that walked around `ForOperation(OpRead)` guards. `OPTIONS` returns `204` with an `Allow` header, and neither logs the spurious "Response step reached with ctx.DBResult == nil" warning that filled logs at probe frequency. **Behaviour change:** `ctx.Operation` for a HEAD request is now `OpRead`/`OpList`; `maniflex.OpHead` is retained but never set ‚Äî scope HEAD-aware middleware with `OpRead`/`OpList`.
- **Bugfix:** a nullable keyset cursor field (e.g. `PublishedAt *time.Time` as `cursor_field`) is now rejected at registration instead of silently breaking pagination ‚Äî the boundary predicate is never true for NULL and the drivers disagree on where NULLs sort (Postgres last on ASC, SQLite first), so rows with a NULL cursor value were dropped or repeated across pages. **Behaviour change:** a model whose `cursor_field` is a pointer field now fails to register; make the column non-nullable or pick another cursor field.
- **Bugfix:** an `in`/`not_in` filter with no values (`?filter=role:in:`) is now rejected with `400 INVALID_QUERY` instead of emitting `role IN ()` ‚Äî a syntax error that any client could use to provoke a 500 on Postgres (SQLite quietly tolerates it). A hand-built `FilterExpr` with an empty value no longer produces broken SQL either: the adapter degrades an empty `IN` to "matches nothing" and an empty `NOT IN` to "matches everything".
- **Bugfix:** an uploaded file's content type is now taken from the part's declared `Content-Type`, falling back to sniffing only when it is absent or `application/octet-stream` ‚Äî sniffing previously overrode the declared type, and since it cannot recognise JSON, CSV, or office formats (they sniff as `text/plain`), a correctly-declared upload to an `mfx:"accept:application/json"` field was rejected with 415 every time.
- **Security:** multipart request bodies are now bounded ‚Äî previously only the in-memory buffer was capped and the overflow spooled to temp files unchecked, so a single long upload could fill the disk. A multipart request over `FilesConfig.MaxUploadBytes` (new, default 32 MB) is rejected with `413 BODY_TOO_LARGE` as it streams, on both model uploads and `POST /files`; `FilesConfig.MaxUploadMemory` (new, default 32 MB) tunes the in-memory buffer, and a per-model `body.MaxBodySize` overrides the ceiling. **Behaviour change:** uploads larger than 32 MB now need an explicit `MaxUploadBytes`.
- **Bugfix:** an oversized JSON body is now rejected with `413 BODY_TOO_LARGE` instead of being silently truncated to the 4 MB limit ‚Äî a padded body whose first 4 MB parsed as complete JSON was previously accepted, creating a record from a body the client never sent. `body.MaxBodySize` now genuinely moves the ceiling (a raised limit was still cut at 4 MB) and reports the 413 it documents; `ctx.BindJSON`/`TryBindJSON` return 413 rather than 400 for the same condition. New `ctx.SetMaxBodySize(limit)` sets the per-request ceiling.
- **Bugfix:** the `mfx:"lock_when"` guard now fails closed ‚Äî a read error other than "record not found" rejects the request (500 `DB_ERROR`) instead of silently letting the update/delete through, and the guard reads through the request's active transaction so it sees uncommitted state (previously it also deadlocked against the request's own transaction on SQLite).
- **Bugfix:** `ModelConfig.OptimisticLock` no longer permits a lost update ‚Äî the `If-Match` check and the write it guards now run as one row-locked transaction, so of two concurrent writers holding the same ETag exactly one succeeds and the other gets its 412 (previously both passed the check and the second silently clobbered the first).

## v0.1.3 (2026-07-06)

- `events.DeliverWithRetry` no longer swallows handler failures silently: it logs a WARN on each retried attempt and an ERROR when all attempts are exhausted (dead-lettering when a DLQ is configured, or "event dropped" when it isn't). Previously a handler that failed every time produced no log at all, so failing events vanished without a trace.
- `maniflex.List[T]` no longer returns zero rows when passed a non-nil `*QueryParams` whose `Limit` is left at its zero value. A hand-built `&QueryParams{Filters: ‚Ä¶}` previously issued `LIMIT 0`; `Limit <= 0` (and `Page <= 0`) now fall back to the default, matching the `q == nil` case. Only the programmatic typed `List` was affected ‚Äî the HTTP list path fills the limit from `?limit`.
- `GET /health` with `HealthCheckDB: true` now reports `db:"ok"` (or `"error"`) even before any model is registered. The ping set previously came only from registered models, so a bootstrapping server with a reachable `Config.DB` but no models yet reported `db:"unknown"`. `Config.DB` is now always included.
- Raw SQL (`ctx.RawQuery`/`ctx.RawExec` and the adapter's `Raw`) now rebinds `?` placeholders to the adapter's dialect, so `?` works on Postgres (`$1, $2, ‚Ä¶`) as well as SQLite. `?` inside string literals is left untouched.
- `ctx.RawQuery` now returns the rows from a data-modifying statement with a `RETURNING` clause (`UPDATE ‚Ä¶ RETURNING`, `INSERT ‚Ä¶ RETURNING`), on both the autocommit and in-transaction paths. Previously query-vs-exec was decided by a `SELECT`-prefix check, so a `RETURNING` statement was routed to `ExecContext` and its result set was silently discarded.
- `mfx:"unique"` is now honoured when the column is added to an existing table via `ALTER TABLE ADD COLUMN`, not only when the table is first created. AutoMigrate creates a `UNIQUE INDEX` (`uidx_<table>_<column>`) on both paths. Previously the constraint was silently skipped (warn only) on the add-column path, so a uniqueness guarantee could vanish on a redeploy that added the column. **Behaviour change:** if existing rows already violate the new constraint, migration now fails with an error naming the table and column instead of skipping ‚Äî resolve duplicate data before deploying.
- Added `auth.JWKSAuth(jwksURL string, opts ...JWTOptions)` for verifying asymmetric JWTs against a rotating JWK Set (an IdP's `/.well-known/jwks.json`). Keys are fetched, cached (1h TTL), and selected by the token's `kid`; an unknown `kid` triggers a rate-limited refetch so key rotation needs no redeploy. Supports RSA (`RS256/384/512`) and EC (`ES256/384/512`), stdlib only. The existing static-key `auth.JWTAuth` is unchanged.
- `jobs/sql`: added `WithTableName` (run a queue on a custom table so two queues can share a DB ‚Äî the table and its indexes are renamed together) and `WithPayloadCipher` (encrypt the payload column at rest; stored values are prefixed `encq:` and decrypted on dequeue). Both are variadic options on `New`/`Migrate`, so existing calls are unchanged.
- `jobs`: a worker that has no handler for a dequeued job's type now requeues it (`Nack`) so another worker can claim it, instead of dead-lettering it. This makes a type-restricted worker safe to run against a table shared with other job types.
- Added `db.RateLimitAction` and `db.AuditLogAction` for custom `server.Action` handlers. The standard `db.RateLimit`/`db.AuditLog` register on the DB step, which actions skip ‚Äî so an all-action service got no rate limiting and no audit records. The `*Action` variants run in the action's own `Middleware` list (after Auth): `RateLimitAction` keys on the caller plus request method+path; `AuditLogAction` records actor/resource/result from the action context. The actions docs now call out that DB-step middlewares don't apply to actions.
- `mfx:"encrypted"` fields are now encrypted/decrypted on the typed access paths (`maniflex.Create`/`Read`/`Update`/`List`) and `ctx.GetModel(name)`, not just the HTTP pipeline. Previously typed/background access went straight to the adapter and stored plaintext at rest (and returned undecrypted ciphertext) ‚Äî the opposite of the intent for a PII column. Background contexts built with `NewBackground` should call `ctx.SetKeyProvider(srv.KeyProvider())`; a new `Server.KeyProvider()` getter exposes it. The raw adapter stays intentionally encryption-agnostic (storing ciphertext verbatim), which is what `RotateEncryptionKey` depends on.
- **Breaking:** relations are no longer inferred from a field's `ID` suffix. A `<Name>ID` field is now a plain scalar column unless you opt in with `mfx:"relation"` (target inferred by stripping `ID`, e.g. `AuthorID` ‚Üí `Author`) or `mfx:"relation:Target"`. This removes the footgun where a value column that merely ended in `ID` (e.g. `CloudEventID`, `ExternalID`) silently became a `BelongsTo`. **Migration:** tag every `<Name>ID` field you want as a relation with `mfx:"relation"`; the `mfx:"norelation"` opt-out is now a deprecated no-op (still parsed so existing models compile). A bare `mfx:"relation"` on a field that doesn't end in `ID` warns at startup (a future strict mode will reject it ‚Äî use `mfx:"relation:Target"`).
- **Behaviour change:** `json:"-"` now hides a field from API responses while keeping it as a real, persisted column (marked read-only so clients can't write it), instead of dropping the column entirely. This matches the Go convention that `json:"-"` affects serialization, not schema. Use `db:"-"` or `mfx:"-"` to exclude a field from persistence. Models that used `json:"-"` to omit a column will now get one on the next migration (AutoMigrate only adds columns, never drops).
- **Breaking:** replaced `Config.AutoMigrate bool` with `Config.DisableAutoMigrate bool`. Migration now genuinely runs by default (the old `AutoMigrate` documented "Default: true" but its zero value was `false`, so a server built from a bare `Config` or `ConfigFromEnv` silently skipped migration and 500'd on the first query). **Migration:** delete `AutoMigrate: true` (now the default); replace `AutoMigrate: false` with `DisableAutoMigrate: true`.
- **Breaking:** the file-upload settings on `Config` are consolidated into a single `Config.FilesConfig` struct. The flat `FileStorage`, `FileSignedURLTTL`, and `FileMiddleware` fields are removed and become `FilesConfig.Storage`, `FilesConfig.SignedURLTTL`, and `FilesConfig.BeforeMiddlewares`. The standalone `/files` endpoints (`POST /files`, `GET /files/*`, `DELETE /files/*`) are also no longer auto-mounted just by configuring a storage backend ‚Äî you must opt in with `FilesConfig.MountEndpoints: true`. Per-model `mfx:"file"` attachment routes are unaffected: they mount off `FilesConfig.Storage` alone and still run the full Auth/DB pipeline. Two new hooks land alongside: `FilesConfig.KeyGen` overrides the storage key derived for a standalone upload (default `DefaultKeyGen` ‚Üí `uploads/<uuid>/<sanitised-filename>`, now exported, you can use it for per-tenant prefixes read from `ctx.Auth` for example), and `FilesConfig.AfterMiddlewares` observe the outcome of a `/files` request for audit/metrics only (the response is already streamed, so read the status via `ctx.Writer.(interface{ Status() int }).Status()`; to alter or block a request use `BeforeMiddlewares`). Mounting the endpoints while `Storage` is still nil is allowed and returns `501 NO_STORAGE` per request. **Migration:** move `FileStorage: fs` into `FilesConfig: FilesConfig{Storage: fs, MountEndpoints: true}` (the `MountEndpoints: true` is required to keep `/files` reachable ‚Äî previously a non-nil `FileStorage` mounted them implicitly); rename `FileSignedURLTTL` ‚Üí `FilesConfig.SignedURLTTL` and `FileMiddleware` ‚Üí `FilesConfig.BeforeMiddlewares`. `Server.SetStorage` is unchanged (it still sets `FilesConfig.Storage`).

Headless models are also excluded from the generated OpenAPI paths (their schema is still emitted).
- Convention foreign keys (`<Name>ID` fields auto-promoted to a `BelongsTo`) now log a warning at startup when the target model was never registered ‚Äî the common case of storing a foreign id by design in a microservice. Tag the field `mfx:"norelation"` to silence it. The relation is still created; this only warns.
- Omitting a JSON / `SQLTyper` column (`LocaleString`, JSON maps/arrays, `money.Amount`, ‚Ä¶) on create no longer fails with a `NOT NULL` violation: the migrator now gives such columns a synthesised empty-container default (`{}` / `[]` / the type's `driver.Valuer` zero, e.g. `0.0000`). Previously the create/update echo went through the map write path, which carried these columns as the stored text; reads already returned structured values, so the two disagreed.
- Fixed `Register`/`MustRegister` panicking ("model must embed BaseModel") when a `ModelConfig` is inlined inside a slice argument. The `ModelConfig`-binds-to-the-preceding-model rule now applies inside flattened slices too, so the documented `MustRegister(domainA.Models(), domainB.Models())` layout ‚Äî where each `Models() []any` carries its own inline `ModelConfig`s ‚Äî works as intended.
- Fixed a silent OR-vs-AND footgun in programmatically built filters: a `maniflex.FilterExpr` left at its zero value (`Group == 0`) is now treated as ungrouped and AND-ed, instead of being OR-ed into "group 0". Multiple hand-built filters (e.g. an ownership-scoped `user_id` + `archived` list) now AND together as intended rather than silently matching extra rows. Internally, OR groups are `Group >= 1` and the URL `?filter[N]=` syntax maps onto `Group N+1`, so the HTTP contract (`?filter[0]=a&filter[0]=b` = OR) is unchanged. Code that built OR groups by hand using `Group: 0` must switch to a positive group index.
- `AutoMigrate` now fails fast with a clear error when a model declares a field whose Go type has no SQL column mapping (e.g. a bare `map[string]any` or `[]string`).
