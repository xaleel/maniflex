# Changelog

## v0.3.2

- **Bugfix:** `jobs/redis` recovers jobs from a crashed worker. It read only `XREADGROUP ">"`, so a job claimed by a worker that died sat in the pending list forever, never redelivered. `Dequeue` now reclaims entries idle past `ReclaimMinIdle` (5m) via `XAUTOCLAIM`, and a new `RenewLease` keeps a live long job from being reclaimed mid-run. The default `ConsumerID` is now unique per process.
- **Bugfix:** `jobs/sql` `GroupKey` (one running job per key) now holds under load. It was enforced only by a `NOT IN (running)` subquery evaluated before the claim's own UPDATE, so one `Dequeue` could start several jobs of a key. The claim now dedups per key with `ROW_NUMBER()`, and a partial unique index on `(group_key) WHERE status='running'` backstops concurrent Postgres claims.
- **Bugfix:** `jobs/sql` on SQLite no longer hands one job to several workers. The claim was an `UPDATE` stamping `lease_until`, then a `SELECT` re-finding those rows by that timestamp ŌĆö not a unique identifier, so two claims in the same clock tick each matched the other's rows. Now one `UPDATE ŌĆ” RETURNING` (SQLite 3.35+). It also stranded jobs as `running` with an attempt spent that nothing ran.

## v0.3.1 (2026-07-20)

- **Breaking / Feature:** `events/nats` honours `Subscription.Group` as a JetStream queue group, so replicas sharing a group share the work ŌĆö what `Group` already meant on Kafka and Redis. It bound a durable with no queue group, which accepts exactly one subscription, so a **second replica was refused** with "consumer is already bound". Existing durables are not reused; see below.
- **Breaking / Bugfix:** `events/nats` durable names are unique. They were `{group}-{sanitised subject}`, and sanitising maps `.` to `_` and `>` to `all`, so `invoice.*` and `invoice.all` produced one name ŌĆö the second subscription refused with `ErrSubjectMismatch`. Now `{group}-{subject}-{hash}`. **Old durables are orphaned:** drain and remove them; new ones start at the stream's default policy.
- **Bugfix:** `events/redis` writes an event's typed and hub streams atomically. They went out on a plain pipeline ŌĆö batching, not a transaction ŌĆö so a connection lost between them left the event on one stream and not the other, typed and wildcard subscribers disagreeing forever. Now one `MULTI`/`EXEC`. `PublishBatch` also swallowed its marshal error, publishing an undecodable payload.
- **Feature:** `events/redis` stream trimming is configurable via `Options.MaxLen` (default 100,000, unchanged; `MaxLenUnlimited` disables). It was a hard-coded constant, and trimming is not free: it deletes the oldest entries without consulting consumer groups, so an entry read but not yet acked is dropped with no error to the publisher and no notice to the consumer. Now documented as such.
- **Bugfix:** the `events/kafka` and `events/redis` read loops no longer hammer an unreachable broker. A failed read retried on a fixed 1s sleep: no growth, no jitter ŌĆö a fleet retries in lockstep, including when the broker returns ŌĆö and blind to ctx, so shutdown waited it out. Now a jittered backoff (`events.ReadBackoff`, 100msŌĆō30s), WARN per retry, ERROR once at the ceiling. Both were silent.
- **Feature:** `events/kafka` supports TLS and SASL. It offered neither, so only plaintext unauthenticated brokers were reachable ŌĆö every managed cluster was out. New `Config.TLS`, `Config.SASL` (any kafka-go mechanism) and `Config.DialTimeout` feed one dialer shared by the writer, every reader and both admin dials ŌĆö securing only some of them fails partway, publishing while topic creation is refused.
- **Bugfix:** `inproc.Publish` no longer grows goroutines without limit. It spawned one per subscriber per event, parked on a semaphore bounding execution but not the queue behind it ŌĆö a slow handler piled them up until the process died (4982 for 5000 publishes). Each subscription now has a bounded queue (`Options.QueueSize`) drained by `Concurrency` workers; `Publish` returns `ErrQueueFull` rather than blocking, and `Emit` logs it.
- **Feature:** `RelayOptions.OrderedByKey` preserves per-aggregate ordering in the outbox. A failed row backs off while later rows keep shipping, so event 2 could reach the broker before the retried event 1 ŌĆö the older state landing last. A row now waits while an older unshipped one shares its key (the event's `Subject`). Opt-in: it costs head-of-line blocking. Consumer-side ordering remains unguaranteed, now documented.
- **Bugfix:** the outbox no longer discards a row when its dead-letter fails. `routeToDLQ`'s result was ignored and `markShipped` ran regardless ŌĆö and since the DLQ rides the broker that just failed every attempt, that is the ordinary shape of an outage. The row is now retained and kept claimable, retrying delivery then dead-letter until one lands. No `DLQType` still drops the row, as documented.
- **Bugfix:** an outbox row whose payload will not decode is no longer a poison pill. `markError` scheduled a retry, but the bytes cannot become valid, so it burned its attempts then sat forever ŌĆö excluded by `claimBatch` and ignored by a sweep that deletes only shipped rows. It is now dead-lettered at once ŌĆö synthesised from the row's own id and `type`, carrying the raw bytes ŌĆö then marked shipped.
- **Bugfix:** `events/redis` now reclaims messages abandoned by a dead consumer. It read only with `XREADGROUP ">"`, so a consumer that died mid-delivery left its messages pending forever, never redelivered. Consumers now run a periodic `XAUTOCLAIM` sweep (`Options.ClaimMinIdle`). The consumer name is stable (hostname+pid), not timestamped ŌĆö which added a permanent group entry per restart.
- **Bugfix:** in-flight event deliveries are no longer lost at shutdown. `events.Emit` published from an untracked goroutine on `context.Background()` ŌĆö unwaited-for, uncancellable ŌĆö and `inproc.Bus.Close` was a no-op, so at-least-once became at-most-once on deploy. Emit now uses `ctx.GoBackground`, which `Server.Shutdown` drains; `inproc.Close` waits on `Options.DrainTimeout` before cancelling.
- **Bugfix:** `events/rabbitmq` no longer dies silently or reports unsent publishes as delivered. A connection drop closed the delivery channel and the consumer returned quietly, leaving the queue unconsumed while the process kept serving. It now logs an ERROR and calls new `Options.OnSubscriptionClosed`; `Publish` waits for the confirm. Still no reconnect ŌĆö documented, not fixed.
- **Bugfix:** `events/kafka` no longer loses in-flight messages under `Concurrency > 1`. Each goroutine committed its own offset, but Kafka commits are cumulative per partition ŌĆö so a fast handler on offset 4 committed while 3 was still running, and a crash there skipped 3 silently. An offset is now held until every earlier one on its partition completes, then committed as one contiguous run.
- **Bugfix:** `events.Emit` with a direct broker bus no longer publishes before the transaction commits. Only an `outbox.Bus` took that path; every other fell through to a fire-and-forget goroutine, so a rollback left subscribers holding an event for a write that never happened. New `ctx.AfterCommit(fn)` defers a side effect to commit, drops it on rollback, and runs inline with no transaction.
- **Bugfix:** `events.Dedupe` no longer loses an event on a transient handler failure. `Seen` records the ID on handler _entry_, so the retry was dropped as a duplicate and delivery reported success ŌĆö never processed, never dead-lettered. The claim is now released when the handler errors, via new optional `events.DedupeReleaser` (both bundled stores implement it). Concurrent dupes still dropped.
- **Security:** `events.Emit` no longer publishes hidden, write-only or encrypted fields. It marshalled the already-decrypted `ctx.DBResult` into `Data`, so every subscriber, broker topic, WS/SSE client and stored `event_outbox.payload` row got password hashes and encrypted plaintext. New `maniflex.RedactRecord` applies the exclusion set and now backs the versioning snapshot too. Keys unchanged.

## v0.3.0 (2026-07-20)

- **Feature:** `?include=` accepts one level of nesting ŌĆö `?include=author.company` loads a relation of the included relation. Asking for `a.b` implies `a`. Each level stays batched, so the cost is one query per node of the tree rather than per row, and nested rows are scoped, decrypted and field-filtered exactly as the first level: a forced filter applies at every level, and `hidden`/`writeonly` fields on the nested model stay out. Depth is capped at two segments (`maniflex.MaxIncludeDepth`) and `?include=a.b.c` is refused with `400`, because the tree is client-supplied and each level multiplies the work. Every segment must name a real relation; a typo is a `400` rather than a silently absent key. `QueryParams` gains `NestedIncludes map[string][]string`; `Includes` keeps its meaning and existing readers are unaffected. **Applies to the JSON response only** ŌĆö typed relation structs are still populated one level deep, so `post.Author.Company` is not filled in by a typed read.
- **Bugfix:** the outbox relayer's dead-letter payload now matches the one `events.DeliverWithRetry` produces. It reused the original event's ID, so a downstream deduper saw the dead-letter as a duplicate of the event already published and dropped it ŌĆö the failure was dead-lettered into nothing, which is the one outcome a DLQ exists to prevent. It now mints a fresh ID, adds `original_id` alongside `original_type`, and logs a failed DLQ publish instead of discarding the error. Original headers were already carried by both paths and are unchanged. New `events.NewID()` exposes the ID format so both paths mint the same thing.
- **Bugfix:** a composite `UNIQUE` violation now names every column it covers. Both driver normalisers returned the first column only, so `UNIQUE(phone_number, owner_id)` reported `field: "phone_number"` ŌĆö blaming one column for a constraint that neither breaks on its own, highlighting the wrong input on a form and asserting something untrue about a value that is fine. `ErrConstraint` gains `Columns []string`; `Column` is unchanged and still carries the first. The `409` emits one `details` entry per column, and for a composite the message names the whole combination. Single-column violations are byte-for-byte as before. The SQLite parser also no longer mistakes the driver's doubled `"constraint failed: "` prefix for a column name, and strips the extended result code at the parser rather than at the consumer.
- **Breaking / Bugfix:** a short list of Latin and Greek stems now pluralise classically ŌĆö `analysis`/`basis`/`axis`/`diagnosis`/`thesis`/`datum`/`medium`/`matrix`/`vertex`/`criterion`/`phenomenon`, plus `quiz` ŌåÆ `quizzes` ŌĆö including inside compound names, so `BloodAnalysis` maps to `blood_analyses` rather than `blood_analysises`. Enumerated, not derived: a blanket `-um ŌåÆ -a` turns `album` into `alba`, and `index` stays `indexes` because that is the plural the database world uses. **Migration:** `pluralize` produces both table names and auto-detected many-to-many relation keys, so a model named after one of these stems changes table _and_ changes the `?include=` key clients use. AutoMigrate never renames, so it would create the new table alongside the old and the data would appear to vanish. Pin the old name with `ModelConfig{TableName: "..."}` before upgrading, or rename the table yourself.
- **Bugfix:** a custom `JWTOptions.Header` now accepts the `Bearer ` prefix. It was stripped only when the header was literally `Authorization`, so pointing `Header` at `X-Auth-Token` and sending `Bearer <token>` ŌĆö the reflex, since it is what the header being replaced requires ŌĆö parsed the scheme as part of the token and answered a flat `401` about the token being invalid rather than about its framing. `Authorization` still requires the scheme. The scheme name is now matched case-insensitively per RFC 7235 on both, and CSRF's bearer-exemption check shares the same parsing so the two cannot disagree about what a bearer header is.
- **Breaking / Bugfix:** a duplicate value now answers `409 CONFLICT` whichever mechanism catches it. `validate.UniqueField` answered `422 VALIDATION_ERROR` while the database's own constraint answered `409 CONFLICT` for the identical error ŌĆö two statuses, two codes, two messages, and two `details` shapes, decided by which noticed first, so a client guarding a field with the middleware had to handle both. `409` won because a duplicate is a conflict with existing state, not malformed input. `details` on the `409` is now an array, matching every other error the framework emits; it was a bare object on that one branch. The extended result code SQLite appends to its message ("email (2067) already taken") no longer reaches the client ŌĆö it was stripped from the field name but not the message.
- **Bugfix:** an unrecognised `mfx:"file_acl:"` value is now a registration error instead of silently becoming `private`. The parser had cases for `signed` and `public` and a default that assigned `private`, so `file_acl:pubic` looked like it failed closed ŌĆö but the field was asked to serve signed or public URLs and served raw storage keys instead, with nothing to say why. `private` gained a case of its own, having previously been produced by the same default arm that swallowed typos.
- **Security:** `db.ForceFilter` now stamps its scope onto creates and updates, not just onto reads. It appended a filter and nothing else, so a create stored whatever the client sent in the scope column: the row landed outside its own author's scope ŌĆö invisible to them on the very next read ŌĆö and because that column is the client's to send, a caller could plant a row directly into someone else's scope, or rewrite the column on update to push a row out of their own. `Tenancy`, `ActionScope` and scoped-singleton provisioning already stamped; `ForceFilter` was the one that did not. Skipped on update for an `immutable` column.
- **Feature:** `maniflex.ProvidesScope()` marks a scope-establishing middleware (`db.Tenancy`, `db.ForceFilter`, `db.ForceFilterVia`) so it is hoisted out of its step and run right after Deserialize. Scope was conventionally registered on the DB step, which is after Validate, so nothing earlier could ask which rows the caller can see ŌĆö a scoped `Singleton`'s row id was unknown during validation and `validate.UniqueField` reported the row as conflicting with itself, refusing every PATCH with `422`. Opt-in: the framework cannot tell a scope provider from any other middleware.
- **Bugfix:** `readonly` and `immutable` no longer strip a value the _server_ stamped via `ctx.SetField` ŌĆö they mean "not from a client", and a middleware injecting a tenant or owner column is not a client. Previously any such middleware running before Validate had its write silently discarded, so the row was stored with the column empty: invisible to the tenant that created it, and visible to every caller whose scope was also empty.
- **Bugfix:** `db.AuditLog(db.WithChanges())` now diffs a scoped `Singleton` against the caller's own row. A singleton has no `{id}` in the URL, so `ctx.ResourceID` holds the `SingletonID` placeholder until the DB step swaps in the real per-scope id ŌĆö and the audit pre-fetch runs before that, so it read nothing and reported every field of the result as newly set, an edit the caller never made. New `ctx.ResolveResourceID()` performs the lookup read-only; it returns `ctx.ResourceID` unchanged for every other request.

## v0.2.6 (2026-07-19)

- **Security:** `ctx.Aggregate` now validates `SortExpr.Direction` against asc/desc instead of concatenating it into the `ORDER BY`. The column side was already checked against the aggregate aliases; the direction was not, so a developer passing user input as `Direction` had an injection point. The HTTP endpoint was never affected ŌĆö it constrains the value already.
- **Security:** `?include=` now decrypts `mfx:"encrypted"` fields on the included relation. An included row reached the serializer without the DB step's decryption pass, so it came back as base64 ciphertext while reading the same model directly returned plaintext.
- **Bugfix:** registration now refuses four shapes that used to fail later or not at all: a pointer embed (nil after scanning, panics on first access), two fields mapping to one DB column (lookups took the first, writes the last), `min:`/`max:` values that are not numbers and empty `enum:` members (parsed as a constraint, enforced nothing), and `readonly,required` (stripped before the required check, so never required).
- **Bugfix:** a value a `min`/`max` bound cannot be measured against ŌĆö a string sent to a numeric field ŌĆö now fails validation instead of skipping the check.
- **Bugfix:** the locale last-resort fallback is now deterministic. It ranged the map and took whatever came first, which Go randomises, so a field with two or more locales outside the request chain changed between identical reads ŌĆö destabilising ETags into spurious `412`s and defeating the response cache. The lexicographically smallest key now wins.
- **Bugfix:** setting an `encrypted`+`unique` field to `""` now clears its `{field}_hmac` companion. The digest of the previous value was left behind, and that column is what uniqueness consults ŌĆö it kept blocking a value the record no longer held.
- **Bugfix:** versioning diffs no longer report phantom changes. Values were compared as `fmt` strings, so JSON columns differed on key order, `time.Time` differed on precision, and `1` vs `1.0` from different drivers read as an edit.
- **Bugfix:** a junction's foreign keys now default to `ON DELETE CASCADE`, so deleting either endpoint takes its link rows with it instead of leaving orphans. An explicit `mfx:"on_delete:"` still wins.
- **Bugfix:** `mfx:"default:"` values are now validated per SQL type before reaching the DDL, rather than numeric and boolean defaults being interpolated verbatim.
- **Breaking / Feature:** many-to-many auto-detection now accepts only models carrying _nothing but_ their two foreign keys. Any model with two BelongsTo to distinct models used to be treated as a join table, so `Order{customer_id, address_id, total}` silently registered a CustomerŌåöAddress relation. A junction that carries columns of its own declares itself by embedding the new `maniflex.JunctionModel` alongside `BaseModel`; `ModelConfig.DisableAutoJunction` opts out entirely.
- **Feature:** `maniflex.JunctionModel` with `mfx:"unique"` emits a `UNIQUE` index over a junction's two key columns, so a duplicate link is refused with `409`, and lets includes collapse repeated pairs. Off by default: a junction carrying its own attributes may legitimately repeat a pair ŌĆö `Enrollment{student_id, course_id, term}` holds one row per term, each with its own `_through` payload.
- **Feature:** `ModelConfig.VersionedRequired` makes a failed history write fail the request. By default it is logged and the primary write still succeeds, which is wrong when history is an audit record: the gap is silent. In a transaction, the primary write rolls back with it.

## v0.2.5 (2026-07-19)

- **Fix:** `maniflex.Create`/`Update` and `ctx.GetModel(...).Create`/`Update` now enforce the model's `enum`, `min` and `max` constraints, which previously ran for HTTP callers only ŌĆö the same model accepted in Go what it answered 422 for over HTTP. A violation errors before the adapter call; pass `maniflex.SkipValidation()` where deliberate. `readonly` stays writable here: it means "not from a client", and a job is not a client.
- **Fix:** a GETŌåÆPATCH round-trip no longer corrupts `mfx:"locale"` fields. Split mode emits `name` plus `name_i18n`; the write side understood neither, so echoing a response back stored a JSON scalar the next read could not parse ŌĆö 500 on the record and on the whole collection. `name_i18n` is now consumed on write and wins, a bare string folds into the effective locale, and a stored scalar resolves with a warning instead of erroring.
- **Fix:** `maniflex.Create[T]` now lets `mfx:"default:"` fire. It wrote every column at its Go value, so the same model created over HTTP and in Go disagreed; a field left at its zero is now omitted from the INSERT when it carries a `default:` tag. Relatedly, `default:` on a pointer field now reaches the schema at all ŌĆö it was emitted only for `NOT NULL` columns and silently dropped.
- **Fix:** `ctx.RecursiveQuery` no longer hangs on cyclic data. A parent chain that loops made the CTE run until the process died; it now tracks visited ids and stops at the repeat, returning each node once. `MaxDepth: 0` also stops meaning unlimited ŌĆö it applies `DefaultRecursiveMaxDepth` (100), and `MaxDepth: -1` opts back into an unbounded walk.
- **Security:** `response.RedactField` now masks exports as well as JSON. An export streams bytes and never builds a `ctx.Response`, so a middleware that masked by editing one silently masked nothing ŌĆö an app hiding `salary` from non-admins served it in full at `/employees/export`. Masking is now _declared_ (`ctx.RedactResponseField`, before `next()`) and honoured by every read path; the export stays row-at-a-time.
- **Security:** the `_through` payload on a many-to-many include now honours the junction model's tags. It was copied verbatim from the junction row, so a `mfx:"hidden"` or `mfx:"writeonly"` column surfaced on every include ŌĆö and the Response step could not catch it, since it passes underscore-prefixed keys through untouched. Encrypted columns and columns the junction model does not declare are omitted too.
- **Security:** forced filters (`db.Tenancy`, `db.ForceFilter`) now scope `?include=` rows, for every relation kind. Includes were fetched with only the soft-delete condition, so the scope stopped at the relation boundary ŌĆö and the FK is the client's to set, so a caller could plant a row inside another tenant's include, or attach another tenant's record to read it. A related model with no such column (a shared lookup) stays unscoped.
- **Security:** `ForOperation(OpList)` now covers `OpExport`, and `ForOperation(OpRead)` covers `OpReadAttachment` and `OpReadHistory` ŌĆö the routes that are the same read in another shape. Scoping written for the obvious operation silently missed the derived one: `db.Tenancy` on `OpList` scoped the list and let `GET /:model/export` return every tenant's rows. One-way ŌĆö `ForOperation(OpExport)` still means export alone.
- **Bugfix (breaking):** an explicit JSON `null` for a non-pointer field is now refused with `422` naming the field, instead of answering two ways for one request: sourced from the typed record it stored `""` with a `200`, sourced from the body map it carried the nil to a `NOT NULL` column and leaked the database's constraint error. Nullability is the Go type ŌĆö `*T` gets a `NULL` column and still accepts `null`.

## v0.2.4 (2026-07-19)

- **Bugfix:** a panicking computed field no longer crashes the process. List rows resolve in worker goroutines, which `PanicRecoverer` does not wrap, so an unchecked `row["id"].(string)` on a null id killed the server from any client on any multi-row page ŌĆö while the same field on a single-row read merely returned `500`. A panic is now logged and its field omitted, as an error already was, per-row and batch alike.
- **Bugfix:** `maniflex.Update[T]` no longer overwrites `created_at`, nor any other `mfx:"readonly"` or `mfx:"immutable"` column. It writes the whole record, so a caller building a fresh struct to change one field stamped the zero time (`0001-01-01`) over the row's real creation date ŌĆö those tags were enforced only by the Validate step, which no typed helper runs. Unset ordinary fields are still blanked: it remains a full replace.
- **Breaking / Security:** version history moved from `GET /{model}_history` to `GET /:model/{id}/history`. The flat routes were unauthenticated and unscoped ŌĆö per-model middleware binds to the parent's name, so `Middleware.Auth` and `db.Tenancy` never covered them. The new route runs the parent's read pipeline, so existing auth and tenancy apply unchanged. See `docs/src/advanced-topics/versioning.md`.
- **Security:** the versioning `snapshot` no longer stores the plaintext of `encrypted`, `hidden` and `writeonly` fields, applying the exclusions the `diff` already used. History rows are built from the decrypted record, so these were written to `{model}_history` as cleartext JSON. **Not retroactive** ŌĆö rows already written keep their plaintext; treat them as exposed and purge or re-key.
- **Breaking / Bugfix:** a `UNIQUE` index `AutoMigrate` cannot create now fails the migration instead of being logged ŌĆö the model declared a constraint the database was not enforcing. A failed **plain** index still only warns. Also fixed: the HMAC index behind `mfx:"encrypted,unique"` was never retried once its column existed. **Upgrade:** duplicates in such a field will now fail the boot; de-duplicate first.
- **Feature:** `maniflex.RequiresField("name")` declares the model fields a middleware gates, so a misspelt field name fails at startup instead of failing open. The name is checked against the model, and the check is exact when paired with `ForModel`. Opt-in: a registration that declares nothing is not checked. See `docs/src/reference/strict-mode.md`.
- **Breaking / Feature:** startup validation and `Config.Strict`. Five mistakes that used to warn and discard what you wrote are now hard failures ŌĆö a misplaced `ModelConfig`, an invalid `mfx:"scheduled"` tag, `mfx:"relation"` on a field not ending in `ID`, and middleware on a step its operations never reach. All report together, before migrations. **Breaks misconfigured apps, deliberately.** See `docs/src/reference/strict-mode.md`.
- **Feature:** soft-delete restore endpoint. `ModelConfig.RestoreEnabled` mounts `POST /:model/{id}/restore`, clearing the delete marker. Off by default, and only for models that soft-delete. It dispatches as `OpUpdate`, so existing update middleware ŌĆö auth, tenancy, force filters ŌĆö governs it; `ctx.IsRestore()` tells the two apart. Adapters opt in via `maniflex.Restorer`. Cascade is not undone.
- **Feature:** token revocation and logout. `JWTOptions.Revoker` makes the JWT middleware consult a blocklist after the signature checks out ŌĆö by `jti` (`401 TOKEN_REVOKED`), or a per-user cutoff for "log out everywhere". Ships `auth.Logout`/`auth.LogoutAll`, an in-memory revoker, and a `middleware/auth/redis` submodule. Lookups fail closed (`503`). Also new: `JWTOptions.VerifyToken`. Both default nil.
- **Feature:** both download routes now honour the HTTP `Range` request header, so an interrupted download can resume and a client can seek into a large file. Satisfiable ranges answer `206` with `Content-Range`; past the end is `416`; malformed and multi-range are served whole. Backends opt in via `maniflex.RangeRetriever` (`LocalStorage` seeks, `storage/s3` pushes the window into `GetObject`).
- **Feature:** `mfx:"relation:Parent;onDelete:cascade|setNull|restrict"` is now enforced. It parsed and was stored but nothing ever read it, so deleting a parent left every child FK dangling. Carried by a real database `FOREIGN KEY ŌĆ” ON DELETE` when neither side soft-deletes, and by maniflex in the delete's own transaction when either does. Note SQLite cannot add FKs to existing tables.
- **Feature:** `mfx:"file,upload:stream"` pipes a multipart upload straight to storage as it arrives, instead of buffering the whole request to the app server's disk first. Opt-in per field; the default multipart path is unchanged. `POST /files` now always streams. `max_size` is enforced mid-stream (`413`, partial object cleaned up), and `FileStorage.Store` gets `meta.Size` 0 and must read to EOF.

## v0.2.3 (2026-07-18)

- **Docs:** a full documentation-correctness sweep against the source ŌĆö ~90 fixes across ~30 pages. Corrected systemic drift in error codes, the v0.1.3 `FilesConfig` refactor and the events-vs-service split; removed documented APIs that never existed (`QueryModelSpec`, `OASContact`, `server.ShutdownContext`, `redis.NewCacheStore`); fixed non-compiling examples and mislabelled packages.
- **Bugfix:** `idempotency.Middleware` now works inside an `ActionConfig.Middleware` list. An action's pipeline skips the Deserialize step that fills `ctx.RawBody`, so the middleware hashed an empty body and the "same key, different body ŌåÆ `422`" guard never fired ŌĆö a client could replay a used key with a different payload. It now reads the body itself via the newly exported `ctx.EnsureRawBody()`.
- **Security:** a `Forced` filter set before the DB step now survives the Deserialize step, which rebuilt `ctx.Query` wholesale and discarded it. That left `jobs/maniflex`'s per-actor scope inert: any authenticated non-admin could list every actor's and every tenant's job statuses, and read any status row by id. Filters parsed from the request still come from the parse alone, so a client cannot smuggle one in.
- **Security:** a storage key is now bound to the principal that mints it, so a leaked key cannot be referenced onto another caller's record. Referencing a key checked only that the object existed, so any authenticated caller could point their row at another tenant's blob ŌĆö and `auto_delete` could then delete it. Keys carry a hash of the caller's scope; a mismatch is `403 FILE_FORBIDDEN`. Older keys are unaffected.
- **Feature:** new `srv.RegisterRollup(ŌĆ”)` maintains a denormalised aggregate column on a parent from its children ŌĆö the column an app otherwise recomputes in every write path and which drifts when one path forgets. Each child write recomputes the affected parent inside the child's own transaction, so deletes, re-parenting and soft-deleted children all fall out correctly. `srv.BackfillRollups(ctx)` reconciles existing rows.
- **Feature:** new `maniflex.FileKeys` is an `mfx:"file"` column holding many storage keys ŌĆö a gallery or attachment set. It stores as a JSON array (JSONB on Postgres), and every rule a single-key field enforces now binds per key: existence, `max_size`, `accept`, `file_acl` signing and `auto_delete` cleanup. ## v0.2.2 (2026-07-16)

- **Security (breaking):** exports no longer hold two copies of the result set, nor run unbounded in parallel. Rows are now serialised one at a time as the writer asks for them (20k narrow rows: CSV 34.3 ŌåÆ 24.2 MB, XLSX 42.3 ŌåÆ 27.2 MB). New `Config.MaxConcurrentExports` bounds the rest. **Behaviour change:** at most 4 exports run at once by default; the next is refused `503 EXPORT_BUSY` with `Retry-After` rather than queued.
- **Performance:** a model's fields and relations are now indexed by name, so `FieldByDBName`/`FieldByJSONName`/`RelationByKey`/`RelationByModel` are constant-time rather than a scan; the validate step's strip pass was quadratic on top of that and is now linear. Lookup goes flat (~39ns at any width) where it was linear (~208ns over 40 fields), so wide models gain most ŌĆö a 40-field model resolves ~5.3├Ś faster.
- **Performance:** removed four pieces of repeated work from the request path: `ctx.QueryParam` parses the query string at most once and lazily; replacing an `mfx:"auto_delete"` file reads the pre-write row once per request rather than once per file field; `cast` dispatches on a type assertion (~15% faster per cell, and retires a latent panic on named types); and `/search`'s `?models=` set is derived at mount.

- **Performance:** the total-count query behind every offset list request is now a plain `COUNT(*)` instead of `COUNT(DISTINCT <table>.id)`. The `DISTINCT` was never needed ŌĆö every join a list query can carry is 1:1, so nothing fans the base table out ŌĆö and it bought only a sort/hash dedup over the whole filtered set per request, which also defeats an index-only count. ## v0.1.5 (2026-07-13)

- **Feature:** new filter operators `contains`, `starts_with`, and `ends_with` take a **literal** value: `%` and `_` in it are escaped (and an explicit `ESCAPE` is emitted, so SQLite and Postgres agree), and all three are case-insensitive. `like`/`ilike` values are patterns ŌĆö `?filter=label:like:50%` matches `500 units` too, and there was no portable way to escape a wildcard ŌĆö so anything a user typed (a search box, a filename, an SKU) belongs in the new operators; `like`/`ilike` are unchanged and remain the right tool when the caller really is writing a pattern. `maniflex.LikePattern` builds the escaped pattern for custom adapters.
- **Bugfix:** `If-Match: *` now does what RFC 9110 says ŌĆö it holds for any existing record ("overwrite whatever is there, but don't create it") instead of being string-compared against the ETag, which returned `412 Precondition Failed` on a resource that plainly existed, the one case the wildcard is defined to accept. A `*` on a missing record still returns `404`, and the row lock is taken either way, so the guarded write stays atomic.
- **Bugfix:** a failed boot no longer orphans `Server.Go` loops and background writes ŌĆö when `ListenAndServe` failed (port in use) the services were stopped but the goroutines they were cancelled alongside were never awaited, and a failed migration or a service that refused to start returned from `Start` without so much as cancelling them (`Server.Go` spawns when you call it, not at `Start`). Every boot-failure exit now runs the same drain the graceful path runs, bounded by `ShutdownTimeout`, so a goroutine is never abandoned mid-write.
- **Bugfix:** `PanicRecoverer` no longer swallows `http.ErrAbortHandler` ŌĆö a handler that panics with it is abandoning its response deliberately (`httputil.ReverseProxy` does this when an upstream dies mid-stream), and net/http's contract is to close the connection silently. The recoverer treated it as a crash: it logged a phantom panic at ERROR and appended a `500 PANIC` JSON envelope to a response that was already half-written and half-sent, so the client received a corrupted body instead of a truncated one. It is now re-panicked, as the stdlib expects.
- **Bugfix:** a client that hangs up mid-request is now recorded as `499` with no body, not `504 TIMEOUT` ŌĆö a disconnect cancels the request context, which the pipeline reported as `"request exceeded the configured query timeout"` (a timeout that need not even be configured) and wrote to a socket nobody was reading, so every abandoned request counted against the server's timeout rate. A genuine server-side deadline still gives `504 TIMEOUT`; the disconnect is logged at `DEBUG` rather than as an error. New `maniflex.StatusClientClosedRequest` (nginx's non-standard 499) names the status for metrics and access logs.
- **Bugfix:** SQLite write connections now open their transactions with `BEGIN IMMEDIATE` (`_txlock=immediate`, added to the write DSN by `sqlite.Open`; a `_txlock` you set yourself is kept). `LockForUpdate`, the `If-Match` check and `mfx:"lock_scope"` are all documented to depend on it, but `Open` never set it: `BeginTx` issued a deferred `BEGIN` and the write-skew protection came only from the write pool being capped at one connection. Two read-then-write transactions now serialise on their own ŌĆö the second waits at `BEGIN` and sees the first one's commit, instead of reading the stale row and then failing with `SQLITE_BUSY` on the way out.
- **Bugfix:** `AutoMigrate` now warns when a live column's type no longer matches its model field (`int` ŌåÆ `string` keeps the old `INTEGER`) ŌĆö the diff only ever added missing columns and warned about extra ones, so a changed Go type was neither detected nor reported and the schema drifted silently until a read or write failed. The column is still never rewritten (that can lose data and locks the table): the warning names the table, column, and both types, and the conversion belongs in an explicit versioned migration. The check is spelling- and dialect-tolerant, so a correctly-migrated table stays quiet.
- **Bugfix:** re-deleting an already-soft-deleted row inside a transaction now returns `404` instead of succeeding ŌĆö the transactional soft-delete omitted the pooled adapter's "not already deleted" guard, so the `UPDATE` still matched the row, re-stamped `deleted_at`/`is_deleted` (losing the original deletion time) and answered `204`. Affects any `DELETE` running under `WithTransaction` and the typed `maniflex.Delete` inside a `Batch`.
- **Bugfix:** `maniflex.TxFromContext` no longer hands back a finished transaction ŌĆö `WithTransaction` cleared `ctx.Tx` on commit but left the same tx behind the context value, so a middleware resuming after its `next()` (an audit hook, a `jobs/sql` outbox enqueue) joined a committed transaction and failed with `sql.ErrTxDone`. Both handles are now cleared on commit _and_ on rollback, so such code falls back to the bare adapter.
- **Bugfix:** an `mfx:"auto_delete"` file's old blob is now deleted only after the update succeeds ŌĆö previously it was removed before the DB write, so a failed update (e.g. a unique-constraint conflict) orphaned the row by deleting a file it still referenced.
- **Enhancement:** `realtime.SSEHandler` always sets header `X-Accel-Buffering: no` as a safe default for Nginx deployments
- **Bugfix:** the Response step no longer panics on a `ctx.DBResult` that a `Replace` middleware can plausibly produce ŌĆö a `ListResult` carrying only `Items`/`Total` nil-dereferenced its absent `Query` or divided by a zero `Limit` (a 500 PANIC on every list request), and a non-pointer record panicked inside `reflect.Value.Elem`. A hand-built `ListResult` now has its pagination defaulted, and a `DBResult` of the wrong shape is reported as `500 INVALID_DB_RESULT` naming the type it got.
- **Bugfix:** `ctx.RawQuery`/`ctx.RawExec` no longer escape an active transaction ŌĆö when the `Tx` cannot run raw SQL they silently fell through to the bare adapter (a different connection, outside the transaction), so a raw write a caller believed was atomic committed on its own and survived the rollback; they now fail with the new `maniflex.ErrRawNotSupportedInTx`. This also affected the framework's own traced transaction: turning on `Config.Trace` pushed every in-transaction raw statement onto the bare adapter, which now participates correctly.
- **Bugfix:** graceful shutdown now runs every phase on one shared budget, as documented ŌĆö the HTTP drain, `Service.Stop` and the `OnShutdown` hook each minted a fresh `ShutdownTimeout`, so they were additive (a slow drain plus a hung stop plus a slow hook could take ~3├Ś the timeout and get SIGKILLed mid-stop) and an explicit `Server.Shutdown(ctx)` deadline was ignored by the service/hook phase entirely. - **Bugfix:** `HEAD` is now the `GET` it mirrors with the body suppressed ŌĆö it runs the same read pipeline, so it returns `404` for a record that doesn't exist (it always answered `200`) and read-scoped middleware (auth, tenancy, soft-delete) now applies, closing an existence oracle that walked around `ForOperation(OpRead)` guards. `OPTIONS` returns `204` with an `Allow` header, and neither logs the spurious "Response step reached with ctx.DBResult == nil" warning that filled logs at probe frequency. **Behaviour change:** `ctx.Operation` for a HEAD request is now `OpRead`/`OpList`; `maniflex.OpHead` is retained but never set ŌĆö scope HEAD-aware middleware with `OpRead`/`OpList`.

- **Bugfix:** an `in`/`not_in` filter with no values (`?filter=role:in:`) is now rejected with `400 INVALID_QUERY` instead of emitting `role IN ()` ŌĆö a syntax error that any client could use to provoke a 500 on Postgres (SQLite quietly tolerates it). The `*Action` variants run in the action's own `Middleware` list (after Auth): `RateLimitAction` keys on the caller plus request method+path; `AuditLogAction` records actor/resource/result from the action context. The actions docs now call out that DB-step middlewares don't apply to actions.
- `mfx:"encrypted"` fields are now encrypted/decrypted on the typed access paths (`maniflex.Create`/`Read`/`Update`/`List`) and `ctx.GetModel(name)`, not just the HTTP pipeline. Previously typed/background access went straight to the adapter and stored plaintext at rest (and returned undecrypted ciphertext) ŌĆö the opposite of the intent for a PII column. Background contexts built with `NewBackground` should call `ctx.SetKeyProvider(srv.KeyProvider())`; a new `Server.KeyProvider()` getter exposes it. The raw adapter stays intentionally encryption-agnostic (storing ciphertext verbatim), which is what `RotateEncryptionKey` depends on.
- **Breaking:** relations are no longer inferred from a field's `ID` suffix. A `<Name>ID` field is now a plain scalar column unless you opt in with `mfx:"relation"` (target inferred by stripping `ID`, e.g. `AuthorID` ŌåÆ `Author`) or `mfx:"relation:Target"`. This removes the footgun where a value column that merely ended in `ID` (e.g. `CloudEventID`, `ExternalID`) silently became a `BelongsTo`. **Migration:** tag every `<Name>ID` field you want as a relation with `mfx:"relation"`; the `mfx:"norelation"` opt-out is now a deprecated no-op (still parsed so existing models compile). A bare `mfx:"relation"` on a field that doesn't end in `ID` warns at startup (a future strict mode will reject it ŌĆö use `mfx:"relation:Target"`).
- **Behaviour change:** `json:"-"` now hides a field from API responses while keeping it as a real, persisted column (marked read-only so clients can't write it), instead of dropping the column entirely. This matches the Go convention that `json:"-"` affects serialization, not schema. Use `db:"-"` or `mfx:"-"` to exclude a field from persistence. Models that used `json:"-"` to omit a column will now get one on the next migration (AutoMigrate only adds columns, never drops).
- **Breaking:** replaced `Config.AutoMigrate bool` with `Config.DisableAutoMigrate bool`. Migration now genuinely runs by default (the old `AutoMigrate` documented "Default: true" but its zero value was `false`, so a server built from a bare `Config` or `ConfigFromEnv` silently skipped migration and 500'd on the first query). **Migration:** delete `AutoMigrate: true` (now the default); replace `AutoMigrate: false` with `DisableAutoMigrate: true`.
- **Breaking:** the file-upload settings on `Config` are consolidated into a single `Config.FilesConfig` struct. The flat `FileStorage`, `FileSignedURLTTL`, and `FileMiddleware` fields are removed and become `FilesConfig.Storage`, `FilesConfig.SignedURLTTL`, and `FilesConfig.BeforeMiddlewares`. The standalone `/files` endpoints (`POST /files`, `GET /files/*`, `DELETE /files/*`) are also no longer auto-mounted just by configuring a storage backend ŌĆö you must opt in with `FilesConfig.MountEndpoints: true`. Per-model `mfx:"file"` attachment routes are unaffected: they mount off `FilesConfig.Storage` alone and still run the full Auth/DB pipeline. Two new hooks land alongside: `FilesConfig.KeyGen` overrides the storage key derived for a standalone upload (default `DefaultKeyGen` ŌåÆ `uploads/<uuid>/<sanitised-filename>`, now exported, you can use it for per-tenant prefixes read from `ctx.Auth` for example), and `FilesConfig.AfterMiddlewares` observe the outcome of a `/files` request for audit/metrics only (the response is already streamed, so read the status via `ctx.Writer.(interface{ Status() int }).Status()`; to alter or block a request use `BeforeMiddlewares`). Mounting the endpoints while `Storage` is still nil is allowed and returns `501 NO_STORAGE` per request. **Migration:** move `FileStorage: fs` into `FilesConfig: FilesConfig{Storage: fs, MountEndpoints: true}` (the `MountEndpoints: true` is required to keep `/files` reachable ŌĆö previously a non-nil `FileStorage` mounted them implicitly); rename `FileSignedURLTTL` ŌåÆ `FilesConfig.SignedURLTTL` and `FileMiddleware` ŌåÆ `FilesConfig.BeforeMiddlewares`. `Server.SetStorage` is unchanged (it still sets `FilesConfig.Storage`).

## v0.1.2 (2026-06-20)

- Docs: corrected the `postgres.Open` examples to the real positional signature `Open(writeDSN, readDSN, registry)` (pool/session tuning via `OpenWithConfig`); documented custom-action behaviours (empty `ctx.Files`, the SQLite single-writer transaction caveat, streaming raw bytes via `ctx.Writer`, emitting events from an action), the `mfx:"file"` pre-uploaded-key existence check, and searching localized (`LocaleString`) content.
- Added `ModelConfig.Headless`: registers a model fully (migration, registry, typed access, relations) but mounts no REST routes, freeing its table path for a custom `server.Action` without the chi route-collision panic. Headless models are also excluded from the generated OpenAPI paths (their schema is still emitted).
- Convention foreign keys (`<Name>ID` fields auto-promoted to a `BelongsTo`) now log a warning at startup when the target model was never registered ŌĆö the common case of storing a foreign id by design in a microservice. Tag the field `mfx:"norelation"` to silence it. The relation is still created; this only warns.
- Omitting a JSON / `SQLTyper` column (`LocaleString`, JSON maps/arrays, `money.Amount`, ŌĆ”) on create no longer fails with a `NOT NULL` violation: the migrator now gives such columns a synthesised empty-container default (`{}` / `[]` / the type's `driver.Valuer` zero, e.g. `0.0000`). Set `mfx:"required"` to opt a column out of the default.
- Database `NOT NULL` violations now return `422 VALIDATION_ERROR` (missing required field) instead of an opaque `500` or a misleading `409`. `maniflex.ErrConstraint` gained a `Kind` field (`unique` / `foreign_key` / `not_null`) and the SQLite/Postgres error normalisers classify accordingly.
- Removed the documented-but-nonexistent `auth.AllowPublicWrite()` from the docs. Public sign-up is achieved by scoping `JWTAuth` away from the operation (auth scoping is inclusion-only), which the auth tutorial, FAQ, and reference now show.
- Create and update responses now serialise JSON / `SQLTyper` columns (`LocaleString`, JSON-backed `sql.Scanner` types) as objects/arrays, matching the read/list responses, instead of returning them as raw JSON strings. Previously the create/update echo went through the map write path, which carried these columns as the stored text; reads already returned structured values, so the two disagreed.
- Fixed `Register`/`MustRegister` panicking ("model must embed BaseModel") when a `ModelConfig` is inlined inside a slice argument. The `ModelConfig`-binds-to-the-preceding-model rule now applies inside flattened slices too, so the documented `MustRegister(domainA.Models(), domainB.Models())` layout ŌĆö where each `Models() []any` carries its own inline `ModelConfig`s ŌĆö works as intended.
- Fixed a silent OR-vs-AND footgun in programmatically built filters: a `maniflex.FilterExpr` left at its zero value (`Group == 0`) is now treated as ungrouped and AND-ed, instead of being OR-ed into "group 0". Multiple hand-built filters (e.g. an ownership-scoped `user_id` + `archived` list) now AND together as intended rather than silently matching extra rows. Internally, OR groups are `Group >= 1` and the URL `?filter[N]=` syntax maps onto `Group N+1`, so the HTTP contract (`?filter[0]=a&filter[0]=b` = OR) is unchanged. Code that built OR groups by hand using `Group: 0` must switch to a positive group index.
- `AutoMigrate` now fails fast with a clear error when a model declares a field whose Go type has no SQL column mapping (e.g. a bare `map[string]any` or `[]string`). Previously such columns were silently dropped, so the model registered and "migrated" successfully while the field was never persisted. Wrap the field in a named type implementing `maniflex.SQLTyper`, or exclude it with the `mfx:"-"` tag.
